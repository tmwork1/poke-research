# コードレビュー 2026-07-04

対象: `git diff origin/master...HEAD`（未push の4コミット、30ファイル/約2200行追加）
内容: Qiita インポーター、AI レビュー(article-ai)パイプライン、catalog/一覧・検索画面、Supabase クライアント初期化のリファクタ。

8つの観点（正確性3・reuse・簡素化・効率・altitude・CLAUDE.md準拠）で候補を洗い出し、実装ファイルを直接読んで裏取りしたうえでの結果。

## 指摘事項（重要度順）

### 1. [Critical] Qiita 一括インポートのループに例外処理がない
`src/lib/importers/qiita.ts:386-420`（`syncQiitaCollection`）

記事ごとに `reviewImportArticle` → `upsertItem` を try/catch なしで `await` している。OpenAI呼び出し失敗やレスポンス不備（`summary`/`reason` 欠落など）が1件でも起きると、そこまでに確定した DB 書き込みは残ったまま例外がそのまま外へ伝播する。呼び出し元 `POST /api/import/qiita`（`src/pages/api/import/qiita.ts`）にも catch がなく、クライアント（`scripts/collect-qiita.mjs`）は 500 しか受け取れず、どこまで成功したか分からない。

**対策案**: ループ内で try/catch し、失敗した記事は `skipped`（理由付き）として結果に積み、バッチ全体は継続させる。

### 2. [High] `OPENAI_MODEL` の既定値表記が `.env.example` と食い違う（対応済み）
`src/lib/importers/article-ai.ts:11`、`.env.example:13`

コード側 `DEFAULT_OPENAI_MODEL = 'gpt5-nano'`（ハイフンなし）と `.env.example` の `OPENAI_MODEL=gpt-5-nano`（ハイフンあり）で表記が割れていた。正しいモデル名は `gpt-5-nano`（ハイフンあり）とのユーザー確認が取れたため、`CLAUDE.md` / `AGENTS.md` / `article-ai.ts` の `DEFAULT_OPENAI_MODEL` を `gpt-5-nano` に修正して統一した（`docs/progress/2026-07-04.md` の「`gpt5-nano` にそろえた」という過去の記述は誤りだったことになる）。

**対応**: `src/lib/importers/article-ai.ts:11` を `gpt-5-nano` に修正し、`CLAUDE.md`・`AGENTS.md` の記載も合わせて修正済み。

### 3. [High] `items`/`search` ページで `sourceId` 未検証、不正値で SSR がクラッシュ
`src/pages/items/index.astro:21`、`src/pages/search.astro:21`

`Number(filters.sourceId)` の結果を検証せず `fetchCatalogItems` に渡している。API ルート（`src/pages/api/items/index.ts:15-18` の `parseOptionalNumber`）は不正値を弾いているが、この2ページは弾いていない。`?sourceId=abc` のような URL では `NaN` が `catalog.ts:137` の `.eq('source_id', NaN)` に渡り、PostgREST がエラーを返して `catalog.ts:161` の `if (error) throw error` で例外化、ページ全体が 500 になる。

**対策案**: API ルートと同じ `parseOptionalNumber` 相当のバリデーションを両ページでも使う（`_shared.ts` 側に共通化してもよい＝指摘10と合わせて対応可能）。

### 4. [Medium] Qiita インポートの `upsertSource`/`upsertItem` に TOCTOU レース
`src/lib/importers/qiita.ts:280-368`

「select してから insert/update」という実装で、`sources.origin_url` にも `items.external_url` にも UNIQUE 制約が無いことを `migrations/001_initial.sql` で確認済み。複数のインポート実行が重なると同じソース／記事の重複行が作られうる。

**対策案**: Supabase の `.upsert(payload, { onConflict: '...' })` に置き換えるか、該当カラムに UNIQUE 制約を追加する。

### 5. [Medium] `syncItemTags` がトランザクションなしで delete → insert する
`src/lib/importers/qiita.ts:208-228`

既存 `item_tags` を先に全削除してから作り直しており、`ensureTags` や後続の insert が失敗すると、対象アイテムのタグが失われたまま残る。

**対策案**: 差分更新にするか、DB 側のトランザクション（RPC化）でまとめて実行する。

### 6. [Medium] `/api/items/[id]` の GET と PUT/PATCH でレスポンス形が食い違う
`src/pages/api/items/[id].ts:22`（GET）と `37-51`（PUT/PATCH）

GET は `fetchCatalogItemById`（`source`/`tags`/`annotations` を含む `ItemDetail`）を返すが、PUT/PATCH は `lib/db.ts` の `updateItem` による素の `Item` 行を返す。同一リソースに対する API 契約が不整合で、GET のレスポンス形を前提にしたクライアントコードが PUT/PATCH の結果を扱うと `source`/`tags`/`annotations` が `undefined` になる。

**対策案**: PUT/PATCH も更新後に `fetchCatalogItemById` で読み直して返すか、GET 側を簡略化するか、方針を統一する。

### 7. [Medium] トップページがエラーハンドリング無しで DB 依存になった
`src/pages/index.astro:7`

以前は外部依存のない静的な `Welcome` コンポーネントだったが、`fetchCatalogOverview()` を try/catch なしで `await` するようになった。`fetchCatalogOverview` は `items`/`tags`/`sources`/`annotations` の4クエリを `Promise.all` しており（`catalog.ts:184-197`）、どれか1つでも失敗（RLS未設定、マイグレーション未適用など）するとトップページ全体が 500 になる。

**対策案**: 主要導線であるトップページだけは try/catch でフォールバック表示を用意する。

### 8. [Low] Cloudflare Workers の `env` をモジュールスコープで読んでいる
`src/lib/supabase.ts:51-56`

ファイル読み込み時点（リクエスト実行コンテキスト外）で `cloudflare:workers` の `env` を参照している。Workers ランタイムでは `env` は基本的にリクエスト処理中のみ安全にアクセスできるため、コールドスタート時のモジュール評価自体が失敗する可能性がある。実害は未確認（本番 `wrangler` 環境での動作確認が必要）。

**対策案**: モジュールスコープの警告チェックを削除するか、`getSupabaseClient()` 呼び出し時（リクエスト内）にのみ行うようにする。

### 9. [Efficiency] Qiita インポートが記事ごとに逐次処理
`src/lib/importers/qiita.ts:386-420`

AI レビューと DB 書き込みを `for` ループで直列に `await` しており、N 件の記事では OpenAI 往復時間が N 倍でそのまま合計される。`POST /api/import/qiita`（`src/pages/api/import/qiita.ts:50`）もこれを同期的に待つため、`pages`/`perPage` を大きく指定すると Cloudflare Worker のリクエストタイムアウトに達するおそれがある。

**対策案**: 記事単位で並列化する、またはインポートをキュー/バックグラウンドジョブに切り出し、API は即座にジョブIDを返す設計にする。

### 10. [Reuse] クエリパラメータの数値パースが複数箇所で重複
`src/pages/api/items/index.ts:15`、`src/pages/api/annotations/index.ts:15`、`src/pages/api/import/qiita.ts:17`、`src/lib/importers/qiita.ts:80`

`parseOptionalNumber` / `parsePositiveInteger` / `normalizePositiveInteger` がほぼ同じ「正の整数かどうか判定してフォールバック」ロジックをそれぞれ個別実装している。

**対策案**: `src/pages/api/_shared.ts` に一本化して各所から import する。指摘3の対応にも流用できる。

## 追記: Qiita 検索条件チューニングのための切り出し検証について

今後 Qiita の検索条件（`query`/`pages`/`perPage`）を試行錯誤する運用が見込まれるが、現状の `syncQiitaCollection`（`src/lib/importers/qiita.ts:370`）は呼び出すたびに「Qiita取得 → 記事ごとにOpenAIレビュー → DB upsert」までフルで実行され、取得結果だけを見る手段がない。条件を変えて試すたびに OpenAI 呼び出しコストと DB 書き込みが発生してしまう。

**推奨**: `syncQiitaCollection` に `dryRun` オプションを追加し、AIレビューとDB書き込み（`upsertSource`/`upsertItem`/`syncItemTags`）をスキップして `fetchQiitaItems` の結果（件数・タイトル・タグ）だけを返せるようにする。本番インポートと同じコードパス（取得・重複排除ロジック）を通すため、チューニング結果が実際のインポート挙動と乖離しない。

別案として `fetchQiitaItems` を直接叩く専用スクリプトも手軽だが、将来ページングや重複排除ロジックが変わったときに検証用コードの更新漏れで本番と挙動がズレるリスクがあるため、将来性を優先するなら `dryRun` オプション方式が望ましい。
