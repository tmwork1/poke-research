# AI採否判定（フィルタ精度）最適化

対象: `src/config/ai-review-prompt.mjs`（`buildSystemPrompt()`、記事取り込み前レビューのsystem prompt）、`scripts/eval/eval-filter.mjs`。

## 背景・目的

収集記事をAI（`src/lib/importers/article-ai.ts` の `reviewImportArticle`）が掲載可否（`accepted`）判定する精度を見直したい、という依頼で着手。作業は `git worktree`（`fix/ai-filter-accuracy`）上で実施した。

## 事前バグ修正: `eval-filter.mjs` のプロンプト抽出が壊れていた

`npm run eval:filter` はシステムプロンプトを `src/lib/importers/article-ai.ts` から正規表現 `content:\s*\n?\s*'([^']*...)'` で直接抜き出す実装だったが、PR #18（トピック設定一元化、[[redistributable-topic-config]]）でプロンプト本体が `src/config/ai-review-prompt.mjs` の `buildSystemPrompt()` に切り出されたため正規表現が一致せず、「(抽出失敗: ファイル構造が変わった可能性)」とだけ出力される状態だった。

`scripts/db/retag-existing-items.mjs` が既に行っているのと同じ方法（`topic`/`buildSystemPrompt` を直接import）に修正し、実際に組み立てられるプロンプト全文が出力されるようにした。

## 現状把握（実データでの分析）

修正後の `eval:filter` をローカルDB（採用330件・棄却118件時点）に対して実行し、出力全件を `content-eval` エージェントに読ませて、現行プロンプトの2条件（(1)主題がポケモン関連か／(2)体験談・エッセイ・攻略情報・ツール配布ページ等ではなく技術的実装・手法を扱っているか）に照らした誤採用・誤棄却の候補を洗い出した。

個別記事ごとの誤判定というより、**同一内容の記事なのに判定が割れる構造的なブレ**が最も顕著だった。

## プロンプト修正（`src/config/ai-review-prompt.mjs`）

上記の分析から見えた根本原因4点をすべてプロンプトに反映した。

1. **境界判断基準の明確化**（条件(1)）: 「キャラクター名・用語を変数名や覚え方の題材として使っているだけ（比喩）」なのか「ポケモン固有のデータ・パラメータ・ルール（種族値、タイプ相性、技構成、特性、乱数処理など）そのものを分析・実装の直接対象にしている」のかで判断する基準を追記。id=138/624・141/627のような温度依存のブレを、境界の言語化によって収束させる狙い。
2. **他分野への応用主張への耐性**（条件(1)）: 「記事が他分野への応用や一般化を主張していても、実際の分析・実装対象がポケモン固有のデータであれば主題外とみなさない」旨を追記。id=618タイプの誤棄却対策。
3. **自サイト内目次ページの明示的除外**（条件(2)）: 除外対象の「他ページへのリンクやコメントの集約ページ」の例示に「同一著者・同一サイト内の連載記事一覧・目次ページ」を追加。id=612タイプの誤採用対策。
4. **比喩エッセイ判定の一般化**（条件(2)）: 従来はKaggleコンペ限定だった「技術語彙が登場すること自体は実装・手法を扱っている根拠にならない」という注意書きを、育成論・キャリア論・組織論の比喩として使うエッセイ・コラム記事全般（「ポケモンに学ぶシステム開発」シリーズ等）に一般化。id=5, 343, 612, 613, 615, 617タイプの誤採用対策。

## 実験3: プロンプトのSTEP形式への全体再構成

ユーザー指示によりモデルは `gpt-5-nano` に戻し、代わりにモデル変更ではなくプロンプト自体の改善で id=612・624/138 を解消する方針にした。サブエージェント経由でFable（`model: "fable"`）にプロンプトエンジニアリングの監修を依頼し、以下の指摘を受けて `src/config/ai-review-prompt.mjs` を全面的にSTEP形式へ再構成した（実行都度の記録は本セクションに追記）。

### Fableの指摘と対応

1. **id=612が誤採用され続ける理由**: 目次ページの除外が「他ページへのリンク集約ページ」という600字超の一文の中の5番目の例示に埋もれており、実質読まれていない。判定対象がページ自身ではなくリンク先シリーズ全体の内容にすり替わっていた。→ 条件(1)より前に独立した **STEP 2（ページ種別チェック）** を新設し、「紹介文が添えられていても」「リンク先の内容で代用禁止」を明記。
2. **id=624/138の不整合の根本原因**: プロンプト内に「クラス設計で学ぶ教材は主題とみなす」「覚え方の題材は比喩」という**矛盾する2つの規則**が併存し、境界事例でどちらを引くかがブレていた。→ **STEP 3（主題判定）** に「実仕様テスト」（分析対象が実際のゲーム仕様・実データか、置き換え可能な架空の数値かで判定）を導入し、矛盾する規則を1本化。few-shot例（`topic.config.mjs` の `aiReview.boundaryExamples`）も追加。
3. その他: confidenceの定義追加、Kaggle警戒文とキャリア論警戒文の統合、タグ規則(c)内の帰属曖昧な文の整理、queryを採否根拠にしない旨の明記。

## 実験4: temperature=0の可否と検証情報の取得漏れ修正

id=613の判定ブレ（実験3参照）を受け、サンプリングのランダム性を`temperature=0`で抑えられないか実APIで確認した。

- `curl`で`gpt-5-nano`に`temperature:0`を指定したところ、`400 invalid_request_error`（`Unsupported value: 'temperature' does not support 0 with this model. Only the default (1) value is supported.`）。**`gpt-5-nano`はtemperature固定（既定値1のみ）で変更不可**と判明。
- 代替として`seed`と`reasoning_effort`パラメータは同モデルで受理されることを確認済み（`seed`はベストエフォートの再現性、`reasoning_effort`は推論の作り込み度合いを上げるパラメータで、揺れの抑制に使える可能性がある）。実際の判定安定性への効果は未検証。
- 副次的に、id=613の「採用への後退」の原因調査中、`retag-existing-items.mjs`がaccepted分岐でreasonをログ出力しない実装漏れを発見（棄却分岐のみreasonを出力していた）。両分岐でreason/confidenceを出力するよう修正し、指示書（CLAUDE.md・docs/reference/scripts.md）にも「実験の記録にはaccepted/rejected問わずreason等を含める」旨を追記した。

## 実験5: confidenceによる事後の強制棄却の検討（不採用）

判定ブレ（id=613等）への対策として、「confidenceが閾値未満なら受け取った後に強制棄却する」案を検討した。保存済みの `eval-filter-output.txt`（採用330件、実験1適用前のデータ）でconfidence分布を集計した。

| confidence | 件数 |
|---|---|
| 0.45〜0.65 | 165件（全体の約半分） |
| 0.66〜0.99 | 165件 |

「confidence<0.7なら強制棄却」という閾値を仮に置くと174件（全体の約53%）が対象になる。一方、これまで確認した誤採用（false positive）のconfidenceは id=61=0.58, id=70=0.65, id=343=0.72, id=612=0.78, id=613=0.75 などで、**正当に採用されている記事（id=22=0.58, id=35=0.65, id=39=0.68, id=95=0.62 など）とほぼ同じ範囲に分布**しており、confidenceは正誤をほとんど分離できていない。単純な閾値では誤採用を狙い撃ちできず、正当な記事を大量に巻き添えにする（gpt-5.4-nano実験2で見た「厳格化による正味の劣化」と同種の問題）。**この案は不採用とし、プロンプト側の判定精度向上（実験1・3）で対応する方針を継続する。**
