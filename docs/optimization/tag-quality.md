# タグ品質最適化

対象: `src/lib/importers/article-ai.ts`（記事取り込み時のAIタグ生成プロンプト）、関連する`tags`/`item_tags`スキーマ。

## 背景・目的

収集記事にAIが自動で付与するタグ（`reviewImportArticle`の`tags`、3〜5個）が、検索・閲覧の役に立つ「必要十分」な状態になっていない疑いがあった。

- 全記事に当たり前に付く「ポケモン」タグは情報価値がないため既に削除済み（`migrations/009_remove_pokemon_tag.sql`, `010_remove_pokemon_tag_nfc.sql`）。プロンプト側でも今後生成しないよう明記済み。
- 残る課題として、ユーザーから「explanationが付いているタグが少なく初心者に優しくない」「プロンプトを改善してタグが必要十分になるように実験して条件を決めてほしい」という2つの依頼があった。本ドキュメントは後者（タグ生成そのものの精度）の対応を整理する。前者（タグ解説`explanation`のカバレッジ改善）は方針未決定のため[残課題](#残課題)に記載する。

## 現状把握（実データ13件で確認した問題）

ローカルSupabase開発スタックの`item_tags`と`items.metadata.ai`（収集時のAI判定結果の生ログ）を突き合わせて確認。

1. **汎用語への逃げ**: 「システム開発」（13件中7件）「設計パターン」「データベース設計」（各4件）が、内容の異なる複数記事（ROMのメモリ改造記事、TiDBハイブリッド検索記事、TypeScript型システム記事など）に判で押したように付与されていた。とくにROMハック記事（id11〜13）は内容的に「設計パターン」「データベース設計」と無関係で、AIが安全策として汎用語に倒れていると判断した。
2. **カタカナ濁点の誤字によるタグ重複**: 「ポケモンカート」というタグが確率計算記事（id10）に付与されていた。収集元Zennのトピックは`["prolog","ポケモン","確率"]`で「カード」系の語はなく、AIが要約文中の「ポケモンカードゲーム」からタグを新規生成する際に「ド」を「ト」に取り違えたものと判明（Unicode正規化の問題ではなく生成テキスト自体の誤字。文字コードを直接確認して濁点なし「ト」であることを確認済み）。`existing_tags`に正しい表記「ポケモンカード」が渡っていたにもかかわらず発生しており、既存タグ再利用ロジックのすり抜けではなく、モデル（`gpt-5-nano`）自身の生成時のトークンレベルの確率的誤りと特定した。

## 「必要十分なタグ」の判断基準

実験を通じて次の基準を決定し、プロンプトに反映した。

- そのタグ単体で検索した際に無関係な記事まで大量にヒットしてしまう一般語（システム開発／設計パターン／プログラミング／開発／実装 等）は使わない。記事固有の技術要素（具体的な言語・フレームワーク・ライブラリ・アルゴリズム・手法名）と、記事が扱うポケモン側の具体的対象（カードゲーム、ROM改造、対戦・育成、図鑑データ等）を優先する。
- 新規タグを作る前に必ず`existing_tags`を確認し、同義・類似・綴りが近い語があれば再利用する（表記ゆれ、カタカナの濁点・半濁点の誤字を含む）。
- タグ数は3〜5個（既存の`MAX_AI_TAGS=5`のまま変更しない）。

## 実験方法と結果

OpenAI課金を避けるため、DB内の実データ（title/summary/収集元タグ）を使い、`article-ai.ts`の`getOpenAIConfig()`/`OPENAI_CHAT_COMPLETIONS_URL`を流用した使い捨てNodeスクリプト（スクラッチパスに作成・実行後削除）で、13件×複数プロンプト案×複数試行をオフライン実験した。

| 案 | 内容 | 汎用語の残存 | カード表記の正答率 |
|---|---|---|---|
| A | 現行プロンプト | 高頻度（7/13等） | カード関連5記事中1記事のみ正（誤字が4記事） |
| B | 汎用語回避の指示のみ追加 | 大きく減少（システム開発0件） | 悪化（カード関連6記事全てで誤字） |
| C | 汎用語のブロックリスト例示＋具体性優先の指示 | Bと同程度 | やや改善（5記事中2記事が正） |
| D | Cに「ポケモンカード」の正しい表記を明示するアンカー文を追加 | ROMハック記事(id11-13)で汎用語が逆に増える回帰あり | 改善（正答率27%→53%、3試行×13件集計） |
| **E** | 汎用語回避とアンカー文を1つの指示ブロックに整理統合し、優先度(a)(b)を明示 | **最良**（「システム開発」はid1除きほぼ消失、「設計パターン」誤用はid11で1/9試行のみ） | **最良**（65%, 11/17、3試行×13件集計） |

E案を採用し、`article-ai.ts`のシステムプロンプトのタグ指示部分に反映した。JSON出力契約（accepted/summary/tags/reason/confidence）、`existingTags`優先ロジック、`normalizeAiTags`/`MAX_AI_TAGS=5`、タグの元表記を保持する`tagLabels`機構は変更していない。

誤字（「ポケモンカート」等）はE案でも完全には根絶できず、残存率は約35%だった。モデル自体の生成傾向に起因するため、プロンプト文言だけでは確率的にしか抑制できないと判断し、[残課題](#残課題)に記録した。

## 副次的に発見・修正したバグ

プロンプト変更の検証中に、`src/lib/importers/common.ts`の`ensureTags`が新規タグ作成時に`tags.display_name`列へ書き込む実装（別セッションで先行実装済みの、タグの元表記を保持する`tagLabels`機能）に対応するマイグレーションが存在しないことを発見した。実際に`display_name`付きでタグをINSERTすると`PGRST204: Could not find the 'display_name' column`で失敗し、新規タグが1件でも生成される収集が全て失敗する状態だった。

ユーザー確認のうえ`migrations/011_add_tag_display_name.sql`（`tags.display_name text`列を追加）を作成し、ローカルSupabaseスタックへ適用。再INSERTの成功と`npm run build`を確認済み。

## 検証

- `npm run build` 成功。
- DB内実データの複製入力によるオフライン再送実験（13件×複数プロンプト案×複数試行）で、プロンプト変更前後のタグ出力を比較。
- `migrations/011`をローカルスタックに適用し、`display_name`付きタグINSERTが成功することを確認。

**未検証**: 実際の収集経路（Qiita/Zenn/note経由の`reviewImportArticle`、本文全文の`bodyExcerpt`を使用）での効果測定。今回の実験はDB保存済みのtitle/summaryを複製した簡易入力によるもの。
