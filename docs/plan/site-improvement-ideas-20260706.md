## 1. 検索・発見性

- [x] **検索結果の関連度順ソート**（優先度: 高 / 工数: 中）— 2026-07-06 実装（タイトル一致3 : 要約一致1 の重み付け、`sort=relevance`）
  現状 `catalog.ts` の検索はキーワード一致後も常に日付順で並ぶため、「一致の強さ」が反映されない。タイトル一致 > 要約一致の重み付けや `pg_trgm` の `similarity()` を使った関連度順を `sort=relevance` として追加する。
- [x] **本文（body）も検索対象に含めるオプション**（優先度: 中 / 工数: 中）— 2026-07-06 実装（`migrations/015` で `items.body` 追加、各インポーターが取得済みの本文を2万字まで保存、検索ILIKEと関連度順スコアリングに body を追加）
  現状の検索対象は `title` / `summary` のみだった。本文ハッシュ用に取得している本文をどこまで保存しているかの確認とあわせて、検索範囲を広げる（またはトグルで選べるようにする）。
  実装メモ: 本文はハイライト表示（`<mark>`）の対象には含めていない。カードには本文が表示されない設計（アイテム詳細ページ廃止済み）なので、ハイライト対象を増やしても閲覧上の利益がなく、長文を毎回ハイライト処理する分のコストだけが増えるため。関連度順ソートのスコアリングでは body 一致を要約一致と同重み（タイトルより弱い）で加点し、スコアリング用の候補プール取得時のみ `body` 列を追加取得する（通常の一覧表示では取得しない）。既存9件の本番アイテムへの遡及バックフィルはスコープ外（別途ユーザー判断）。

| 1 | 本番アイテム数の原因調査 + 過去記事バックフィル | 品質 | 記事が少ないままでは情報ハブとしての価値が立ち上がらない（ユーザー報告あり） |
| 2 | CI build 恒常失敗の修正 | 運用 | 他のすべての変更の検証基盤が壊れている |
| 3 | 収集ジョブの失敗通知 | 運用 | 収集停止に気づけないと情報ハブとして空洞化する |
| 4 | AI要約の簡潔化（常体・2行程度） | 品質 | ユーザー要望。プロンプト変更は小さく、要約再生成はバックフィルと同時に実施できる |
| 5 | タグ近似マッチング + 誤字タグ遡及修正 | 品質 | 既知の残存バグで、検索・タグクラウドの信頼性に直結 |
| 6 | fetchTopTags / fetchBookmarkCounts のDB側集計化 | 性能 | アイテム増で確実に悪化する構造的問題。早いほど安い |
| 7 | ブックマーク失敗時のフィードバック | UX | 無言で失敗する操作はユーザーの信頼を削る。修正も小さい |
| 8 | 検索の関連度順ソート | 検索 | 「探せること」は本サイトの中核価値 |
| 9 | About ページ + AI要約の明示 | 信頼 | provenance 重視の方針をUIでも体現する |
| 10 | OG画像PNG化 + JSON-LD | SEO | 共有・流入の入口を整える。工数小 |
