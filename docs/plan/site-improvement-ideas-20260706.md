## 1. 検索・発見性

- [x] **検索結果の関連度順ソート**（優先度: 高 / 工数: 中）— 2026-07-06 実装（タイトル一致3 : 要約一致1 の重み付け、`sort=relevance`）
  現状 `catalog.ts` の検索はキーワード一致後も常に日付順で並ぶため、「一致の強さ」が反映されない。タイトル一致 > 要約一致の重み付けや `pg_trgm` の `similarity()` を使った関連度順を `sort=relevance` として追加する。
- [ ] **本文（body）も検索対象に含めるオプション**（優先度: 中 / 工数: 中）
  現状の検索対象は `title` / `summary` のみ。本文ハッシュ用に取得している本文をどこまで保存しているかの確認とあわせて、検索範囲を広げる（またはトグルで選べるようにする）。
- [ ] **検索キーワードのハイライト表示**（優先度: 中 / 工数: 小）
  検索結果カードのタイトル・要約内で一致箇所を `<mark>` 強調し、なぜヒットしたか分かるようにする。
- [ ] **0件時のフォールバック提案**（優先度: 中 / 工数: 小〜中）
  「一致するアイテムがありません」だけで終わっている。条件を1つ緩めた場合の件数提示、近い綴りのタグ候補、人気タグへの導線を出す。
- [ ] **タグの複数選択（AND / OR 検索）**（優先度: 中 / 工数: 中）
  現状タグは1つしか選べない。`?tag=a&tag=b` を受けてAND絞り込みできるようにする。
- [ ] **期間フィルタ**（優先度: 低 / 工数: 小）
  「直近1ヶ月 / 半年 / 1年」などの公開日フィルタを検索ページに追加する。
- [ ] **タグ一覧ページ（/tags）の新設**（優先度: 中 / 工数: 小）
  現在 `/tags/[name]` はあるが全タグを見渡せるインデックスページが無い。使用回数付きの全タグ一覧はSEO面（内部リンク）でも有効。
- [ ] **検索サジェスト / オートコンプリート**（優先度: 低 / 工数: 中）
  検索ボックス入力中にタグ名・頻出キーワードを提案する。
- [ ] **関連記事の表示**（優先度: 中 / 工数: 中）
  データモデルに `item_relations` があるがUIで一切使われていない。タグ重複ベースの簡易「関連記事」をカード下部やタグページに出すところから始める。
- [ ] **保存済み検索・新着通知**（優先度: 低 / 工数: 大）
  ログインユーザーが検索条件を保存し、合致する新着が入ったら通知（まずはマイページ上のバッジ程度から）。

## 2. UI / UX

- [ ] **ダークモード対応**（優先度: 中 / 工数: 中）
  `Layout.astro` が `color-scheme: light` 固定。`.sort-select` や `.search-input` に `background: #fff` のハードコードが複数あり、まずCSS変数へ寄せてから `prefers-color-scheme` 対応する。
- ~~カードの複数カラム表示~~ → **見送り（決定済み）**
  2026-07-06 のユーザー判断により不採用。「1カラムが最も見やすい（Qiita も同様のレイアウト）」ため、トップ・検索・マイページとも1カラム固定を維持する。今後も再提案しない。
- [ ] **要約の行数制限（line-clamp）**（優先度: 中 / 工数: 小）
  要約の長さで各カードの高さがばらつく。3〜4行でクランプし「続きは元記事で」に誘導する。
- [ ] **タグ用語解説ツールチップの改善**（優先度: 中 / 工数: 中）
  現状は `title` 属性への遅延セットのため、(1) 初回ホバーでは間に合わず表示されないことがある、(2) タッチデバイスでは全く使えない。ポップオーバー実装に置き換え、タップでも開けるようにする。
- [x] **ブックマーク失敗時のユーザー向けフィードバック**（優先度: 高 / 工数: 小）— 2026-07-06 実装（トースト表示）
  `ItemCard.astro` のトグル失敗は `console.error` のみで画面には何も出ない。トースト等で「保存に失敗しました」を表示する。
- [ ] **無限スクロールの状態保持**（優先度: 中 / 工数: 中）
  トップの無限スクロールは戻る操作で先頭に戻ってしまう。読み込んだページ番号をURL（`history.replaceState`）へ反映し、記事リンクから戻った際の位置を復元する。
- [ ] **トップと検索ページのページング方式の不統一を解消**（優先度: 低 / 工数: 小〜中）
  トップは無限スクロール、/items はページネーションと二方式が混在。どちらかに寄せるか、意図的な使い分けとして文書化する。
- [ ] **ローディングスケルトン**（優先度: 低 / 工数: 小）
  無限スクロールの「読み込み中...」テキストをスケルトンカードにする。
- [ ] **モバイルでのログアウト導線の確認**（優先度: 低 / 工数: 小）
  ヘッダーの `account-dropdown` は hover / focus-within 依存でタッチでは開けない（サイドバー側にログアウトがあるため致命的ではないが、挙動として不整合）。
- [ ] **サイドバーのフォーカス制御**（優先度: 中 / 工数: 小）
  閉じたサイドバーは `transform` で画面外にあるだけでリンクがタブ順に残る。`inert` 属性の付け外し、開時のフォーカス移動、閉時のトグルへのフォーカス返却を実装する。
- [ ] **無限スクロール追加分のスクリーンリーダー通知**（優先度: 低 / 工数: 小）
  追加読み込みされたカードが支援技術に伝わらない。`aria-live` リージョンで「N件追加されました」を通知する。
- [ ] **モバイルでの共有に Web Share API を使う**（優先度: 低 / 工数: 小）
  カードの共有ボタンはX固定。`navigator.share` が使える環境ではOSの共有シートを開き、あわせて「リンクをコピー」も足す。
- [ ] **共有ボタンのアイコン見直し**（優先度: 低 / 工数: 小）
  「Xで共有」のラベルに対しアイコンが汎用シェアアイコンで、サイドバーの「このサイトを共有」と同じ図柄。X向けであることが分かるアイコンにするか、共有先を選べるUIにする。
- [ ] **ソースの視覚的識別**（優先度: 中 / 工数: 小〜中）
  カード上のソース表示（Qiita / Zenn / note / ブログ）がテキストのみ。ソース別の色チップやファビコン表示で一覧を眺めたときの識別性を上げる。ブログ記事はドメイン名の表示も検討する。
- [ ] **総件数の表示**（優先度: 低 / 工数: 小）
  検索結果に「全N件」が出ていない（`pagination.total` は既に渡っている）。フィルタ操作の手応えとして表示する。
- [ ] **トップの空状態メッセージの見直し**（優先度: 中 / 工数: 小）
  `index.astro` の空状態が「scripts/db/seed-sample-article.mjs でサンプルを投入できます」と開発者向け文言のまま本番に露出する。タグフィルタで0件の場合と初期状態とで文言を分け、一般利用者向けにする。
- [ ] **既読管理**（優先度: 低 / 工数: 中）
  ログインユーザー向けに「読んだ」マークを付けられるようにし、一覧で薄く表示する。

## 3. コンテンツ品質・収集パイプライン

- [x] **本番アイテム数の少なさの原因調査と過去記事バックフィル**（優先度: 高 / 工数: 中）— 2026-07-06 調査完了・バックフィル準備完了（実行は未・ユーザー確認待ち）。本番は items 9件（Qiita 4 / Zenn 5 / note 0、blog ソース行なし）。主因は増分収集前提の既定値（Qiita 1×20、note 1×10、キーワード2語）+ AIレビュー棄却。キーワードを6語へ拡充し、`npm run collect:backfill`（scripts/collect/backfill.mjs、棄却理由の内訳も出力）を追加。
  本番で一度収集を実行したが DB のアイテム数が非常に少ない（2026-07-06 ユーザー報告）。現行の既定値は「毎日の増分収集」前提で、1回の実行あたり Qiita 1ページ×20件・note 10件、キーワードは「ポケモン」「ポケカ」のみ（`keywords.ts`）、さらに AI レビューの2条件判定で相当数が棄却されるため、初回実行では蓄積がほぼ得られない構造になっている。まず fetched / inserted / skipped と AI 棄却数の内訳を確認して主因を特定し、`pages` / `perPage` を引き上げた過去分の一括バックフィルを手動実行する（OpenAI 課金・各 API のレートリミットに注意）。あわせて AI 判定基準が厳しすぎないかの確認と、検索語彙の拡充（例: 「ポケモンGO」「ダメージ計算」など）も検討する。
- [x] **AI要約の簡潔化（常体・2行程度へ圧縮）**（優先度: 高 / 工数: 小）— 2026-07-06 プロンプト変更済み（常体・2文以内・120字程度）。既存9件の要約はバックフィル実行時に再生成される。
  現状の要約はですます調でやや冗長（2026-07-06 ユーザー要望）。`article-ai.ts` のシステムプロンプトは「日本語で3行以内」としか指定していないため、「だ・である調または体言止めで2行・120字程度」に変更する。既存アイテムの要約は再タグ付けバックフィル（下記）と同じ経路で再生成できるため、実施する場合はまとめて行うと課金が一度で済む。
- [x] **タグの近似マッチングによる誤字タグ根絶**（優先度: 高 / 工数: 中）— 2026-07-06 実装（`ensureTags` 前段で編集距離1の既存タグへマージ。日本語含む5文字以上限定、数字違いは温存）
  「ポケモンカート」問題（2026-07-06 進捗参照）。プロンプト改善で発生率は下げたが根絶できていない。タグ保存前にレーベンシュタイン距離等で既存タグと照合し、閾値内なら既存タグへマージする仕組みを `common.ts` の `ensureTags` 前段に入れる。
- [x] **既存の誤字タグ「ポケモンカート」の遡及修正**（優先度: 高 / 工数: 小）— 2026-07-06 確認の結果、本番DBに「ポケモンカート」タグは存在しなかった（全26タグを確認）。今後の遡及修正用に `scripts/db/merge-tag.mjs <from> <to>`（冪等）を追加。
- [ ] **既存アイテムの再タグ付けバックフィル**（優先度: 中 / 工数: 中）
  改善後プロンプトは今後の収集にしか効かない。既存アイテムへ新プロンプトを再適用するスクリプトを用意する（OpenAI課金が発生するため件数と実行タイミングはユーザー判断）。
- [ ] **タグAI解説（explanation）のバックフィル方針決定**（優先度: 中 / 工数: 小〜中）
  28件中7件にしか解説が無い問題が保留中。バックフィルスクリプト / 新規タグ作成時の自動生成 / 表示箇所拡大の3案から決めて実施する。
- [ ] **重複記事の検出**（優先度: 中 / 工数: 中）
  同一記事の Qiita / Zenn / 個人ブログへのクロスポストが別アイテムとして並ぶ可能性がある。タイトル類似度や正規化URLで検出し、`item_relations` で「同一内容」として束ねる。
- [ ] **リンク切れの定期検出**（優先度: 中 / 工数: 中）
  元記事の削除・非公開化を検出する仕組みが無い。cron で外部URLを定期チェックし、404 が続くアイテムに印を付ける（または一覧から隠す）。
- [ ] **記事更新の検知と表示**（優先度: 低 / 工数: 中）
  本文ハッシュ（`items.version`）で差分検知はしているが、UI上「更新された記事」であることが分からない。「更新あり」バッジや更新日時の表示を検討する。
- [x] **AI要約であることの明示**（優先度: 中 / 工数: 小）— 2026-07-06 実装（カード要約に「AI要約」バッジ）

  ロードマップ（M3）で「リリース後アップデート」とされている項目。記事単体URLの投稿フォーム + 既存のAIレビュー経路への接続 + 管理者承認キューとして設計する。
- [ ] **収集ソースの追加検討（はてなブックマーク等）**（優先度: 低 / 工数: 中〜大）
  はてなブックマークの人気エントリ検索、Speaker Deck / Docswell のスライドなど。GitHub / YouTube を対象外とした既存の判断（ロードマップ M3）は維持する。
- [ ] **タグへのユーザーフィードバック**（優先度: 低 / 工数: 中）
  「このタグは不適切」を報告できる小さな導線を付け、タグ精度改善のデータにする。

  正規化（normalizeItem 等）・URL構築・ソート分岐はDBなしでテスト可能。回帰防止として最低限を書く。
- [ ] **死活監視**（優先度: 低 / 工数: 小）
  外形監視（UptimeRobot 等）でトップと `/api/items` を監視し、Cloudflare / Supabase 障害に気づけるようにする。

## 7. コード整理（小粒）

- [ ] **CatalogPage の未使用 `title` prop の削除**（優先度: 低 / 工数: 小）
  見出し廃止後も `title` prop が残っており、呼び出し側から渡されているが描画されない。インターフェースから落とす。
- [ ] **ソート select 等の重複スタイル・スクリプトの共通化**（優先度: 低 / 工数: 小）
  `.sort-select`（データURI背景含む）と submit スクリプトが index / mypage / CatalogPage に3重に重複している。`SortSelect.astro` として切り出す。
- [ ] **`fetchCatalogOverview` など未使用コードの棚卸し**（優先度: 低 / 工数: 小）
  アイテム詳細画面の廃止で使われなくなった関数（`fetchCatalogItemById` 等も要確認）を整理する。

## 優先着手の推奨（上位10件）

| # | 項目 | 区分 | 理由 |
|---|------|------|------|
| 1 | 本番アイテム数の原因調査 + 過去記事バックフィル | 品質 | 記事が少ないままでは情報ハブとしての価値が立ち上がらない（ユーザー報告あり） |
| 2 | CI build 恒常失敗の修正 | 運用 | 他のすべての変更の検証基盤が壊れている |
| 3 | 収集ジョブの失敗通知 | 運用 | 収集停止に気づけないと情報ハブとして空洞化する |
| 4 | AI要約の簡潔化（常体・2行程度） | 品質 | ユーザー要望。プロンプト変更は小さく、要約再生成はバックフィルと同時に実施できる |
| 5 | タグ近似マッチング + 誤字タグ遡及修正 | 品質 | 既知の残存バグで、検索・タグクラウドの信頼性に直結 |
| 6 | fetchTopTags / fetchBookmarkCounts のDB側集計化 | 性能 | アイテム増で確実に悪化する構造的問題。早いほど安い |
| 7 | ブックマーク失敗時のフィードバック | UX | 無言で失敗する操作はユーザーの信頼を削る。修正も小さい |
| 8 | 検索の関連度順ソート | 検索 | 「探せること」は本サイトの中核価値 |
| 9 | About ページ + AI要約の明示 | 信頼 | provenance 重視の方針をUIでも体現する |
| 10 | OG画像PNG化 + JSON-LD | SEO | 共有・流入の入口を整える。工数小 |
