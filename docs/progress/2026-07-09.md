# 進捗メモ 2026-07-09

## 新着記事のX投稿下書きをDiscordに通知（`feat/new-items-x-draft-digest`）

cronで新着記事（AIレビューで採用され新規insertされた記事）が見つかったとき、Xに投稿しやすい形式の下書き文をまとめてDiscordへ送る機能を追加。

- 対象は収集系cronジョブ全て（Qiita/Zenn/note/Blog(Brave Search)/RSSフィード追従/はてなブックマーク）。link-check・週次DBレビューは対象外（新着記事を扱わないため）。
- 下書きは新規のOpenAI呼び出しを追加せず、AIレビュー結果とは独立に `サイトURL＋各記事のタイトル/URL` だけで組み立てる（後述の140字制限対応の経緯を参照）。
- 送信先は既存の `ALERT_WEBHOOK_URL`（運用アラート・週次レポートと同じDiscordチャンネル）をそのまま使用。新しい環境変数は追加していない。
- 複数件見つかった場合は1件ずつのメッセージではなく、まとめて1つのDiscordメッセージに列挙する。

### 実装
- `src/lib/notify.ts`: `sendNewItemsDigest()` を追加。`action==='inserted'` の記事から下書きを組み立て、Discordへ1メッセージで送る。
- `src/worker.ts`: 各cronハンドラ（Qiita/Zenn/note/Blog/Feed/Hatena）の成功ログ直後に、`action==='inserted'` の記事があれば `sendNewItemsDigest` を呼ぶ `notifyNewItems()` ヘルパーを追加。

初版はAIレビュー生成の要約80字＋タグ由来ハッシュタグを記事ごとに付けていたが、Xの140字制限では
新着が2件以上のとき収まらないとユーザー指摘があり修正。要約・ハッシュタグは廃止し、下書きは
`topic.site.url`（PokeResearchのサイトURL）＋各記事のタイトル/URLのみのシンプルな構成にした
（`process-import-item.ts` へ一度追加したsummary/tagsの受け渡しも、不要になったため差し戻した）。

### バグ修正: `notify.ts` のDiscord判定が `discordapp.com` 形式のWebhook URLに非対応だった
ユーザーの `.env` へ一時的に本番 `ALERT_WEBHOOK_URL` を追加してもらい、`sendNewItemsDigest` を
ダミーデータで実際に呼び出したところ、Discordが400を返した。原因は `postToWebhook` の
`isDiscord` 判定が `discord.com/api/webhooks` の部分一致のみで、旧ドメイン
`discordapp.com/api/webhooks`（今回のWebhook URLの形式）にマッチしなかったこと。Slack形式
`{text}` ペイロードがDiscordに送られて拒否されていた。`/discord(app)?\.com\/api\/webhooks/` に
広げて修正（`sendOperationalAlert`/`sendMaintenanceReport` も同じ関数を使うため、このバグは
以前から運用アラート・週次レポート全体に影響していた可能性がある）。

### メッセージ形式をユーザー指定に合わせて調整
実送信の見た目を見たユーザーから指定があり、絵文字プレフィックス（🆕）とジョブ名（「Qiitaで」等）
を外し、`[poke-research] 新着 N 件` のヘッダー＋`サイトURL（末尾スラッシュ付き）＋記事タイトル/URL`
だけのシンプルな形式に変更。`sendNewItemsDigest` からジョブ名の引数を削除し、`worker.ts` 側の
呼び出し（`notifyNewItems`）も追随。

### フォーマット再調整：記事ごとに掲載元（ソース）名を付与
ユーザー指定で最終形式を `【poke-research】新着 N 件` ＋ `サイトURL` ＋ 各記事 `タイトル - ソース名`／
`URL` に変更。Qiita/Zenn/noteは掲載元が固定だが、Blog(Brave Search)/RSSフィード/はてなブックマーク
は記事ごとに実際の個人ブログ等が異なるため、ジョブ名を決め打ちにできない。新規に
`fetchItemSourceNames()`（`src/lib/importers/common.ts`）を追加し、`items.source_id` 経由で
`sources.name` を都度引くようにした。JOINの正規化は `catalog-normalize.ts` の既存
`normalizeSource`（Supabaseの結合結果が単一オブジェクト/配列どちらでも返りうる問題への対処）を
そのまま再利用している。`worker.ts` の `notifyNewItems()` が新着itemのidから逆引きして
`sendNewItemsDigest` に渡す。

## タグクラウードのkind対応・cronへのarXiv組み込み（`feat/paper-collection`、上記2件のフォローアップ）

### タグクラウードのkind対応
- `migrations/023_top_tags_kind_filter.sql`（新規）: `top_tags` RPC（migrations/012）に
  `kind_filter text DEFAULT NULL` を追加し、`item_tags` → `items` へJOINして
  `kind_filter IS NULL OR i.kind = kind_filter` で絞り込む。引数を1個→2個に増やす変更は
  `CREATE OR REPLACE` だけでは別シグネチャの関数として並存し、デフォルト引数を持つ新シグネチャが
  既存の1引数呼び出しと曖昧になって `function is not unique` エラーになるため、まず
  `DROP FUNCTION IF EXISTS top_tags(integer)` してから新シグネチャを作成する形にした。
  `GRANT EXECUTE` も新シグネチャ（`top_tags(integer, text)`）に対して再実行。可視条件
  （`link_status`/`ai_accepted`）は既存の `top_tags` 自体が課していないため、今回も揃えて追加は
  していない（挙動を変えすぎない方針）。
- `src/lib/catalog.ts`: `fetchTopTags(limit = 20, kind?: string)` に第2引数を追加し、
  `rpc('top_tags', { tag_limit, kind_filter: kind ?? null })` を渡す。
- `src/pages/items/index.astro` は `fetchTopTags(20, 'article')`、`src/pages/papers/index.astro` は
  `fetchTopTags(20, 'paper')` に変更。HOME（`src/pages/index.astro`）・`src/pages/tags/index.astro`・
  `src/pages/tags/[name].astro`・`src/pages/sitemap.xml.ts` は企画上、記事・論文を横断した全体
  タグクラウードのままでよいため変更していない。
- 検証: CIと同じ使い捨てPostgres（Docker `postgres:16`、`auth.users` スタブ込み）を起動し、
  `001`〜`023` を `npm run migrate` で空DBに順次適用して green を確認。さらに `items`/`tags`/

### 実行時間についての所見（CPU時間 vs 実行時間）
6ジョブ（元々30分間隔で6回に分散していたもの）を1回のCron Trigger発火に集約したことで、
Cloudflare Workersの制限に抵触しないか確認した。
- **CPU時間**: 各ジョブの処理はOpenAI API呼び出し・外部サイトへのfetch・Supabaseへのクエリなど
  I/O待ちが大半で、実際のCPU使用時間（Workers課金・制限の対象）はJSON/XMLパースや文字列処理程度に
  留まる見込み。6ジョブを1回にまとめても、待機時間はCPU時間に計上されないため、Workersプランの
  CPU時間制限（既定は数十秒〜、`wrangler.jsonc`に`limits.cpu_ms`の指定は無く現状デフォルトのまま）
  に抵触する可能性は低いと判断した。
- **実行時間（wall-clock）**: Cloudflare の Cron Trigger は `scheduled()` ハンドラ・
  `ctx.waitUntil()` に渡した処理を合わせて一定時間内（目安として最大15分程度）に完了する必要が
  あるとされ、6ジョブの合計時間がこの範囲に収まるかは実測していない。以前は30分間隔で分散して
  いたため個々のジョブの所要時間が長くても影響が出にくかったが、集約後は「6ジョブの合計」が
  1回の発火の時間予算に収まる必要がある。深刻な懸念とまでは判断せず実装を進めたが、
  本番適用・デプロイ後は初回数回の実行時間をCloudflareダッシュボード/ログで確認し、予算に近い
  場合はBrave Search（ページ数が多いと時間がかかりうる）等の重いジョブを見直すか、
  アカウント全体で cron trigger にまだ2枠の余裕がある（現在3/5使用）ため、必要なら再度スロットを
  分割する対応を検討したい。

## 日次収集Discord通知の0件対応（PR #36）

これまで日次収集ジョブ（Qiita/Zenn/Blog/はてな/arXiv/フィード、計6ソース）は各ジョブが個別に
`sendNewItemsDigest` でDiscordへ通知しており、新着0件のソースは何も送信しない仕様だった。
そのため「cronが正常に実行されたかどうか」を通知だけでは確認できないという課題があり、
ユーザー指示で以下に変更した。

- `src/lib/notify.ts`: `sendNewItemsDigest`（ソース別即時通知、0件は送信しない）を廃止し、
  `sendDailyDigest(env, items, breakdown)` に置き換え。`breakdown`（ソース名と件数の配列）を
  ヘッダーに含め、`合計0件でも必ず送信する`（cronの生存確認シグナルとして機能させる）。
- `src/worker.ts`: `runScheduledXxxImport` 系の各関数を `Promise<void>` から
  `Promise<ImportItemOutcome[]>` に変更し、ジョブ内で個別に通知していたのをやめて結果を返すだけに
  した。`runDailyJobsSequentially()` が全ジョブの結果を集約し、ループ完了後に
  `notifyDailyDigest()` を1回だけ呼ぶ構成に変更。リンク切れ検出は「新着」の概念がないため集約対象
  （`DAILY_COLLECTION_JOBS`）に含めず、元の実行順（ブログの後・はてなの前）を保ったまま個別に呼ぶ。
- ローカルの`ALERT_WEBHOOK_URL`（本番とは別のテスト用チャンネル）に実際に0件Digestを送信し、
  期待通りのメッセージ（`【poke-research】本日の収集結果: 合計0件（フィード 0件・Qiita 0件・...）`）
  が届くことを確認したうえでmainへマージ。

実際にCloudflare Cron Triggerで動くかを本番で確認したところ（PR #37で発火時刻を一時変更）、
日次収集ジョブを1回のWorker呼び出しに集約する既存設計がCloudflareのsubrequest数上限と衝突し、
チェーンが途中で失敗する重大な問題が見つかった。発火時刻は`0 0 * * *`（JST 09:00）へ復旧
（PR #39）。詳細な調査結果・再現手順は
[docs/issue/cron-subrequest-limit.md](../issue/cron-subrequest-limit.md) を参照。

## cron subrequest上限問題の恒久対応: Qiita/Zenn/arXivへの差分検知移植（`fix/cron-subrequest-diff-detection`）

上記issueの根本対応。コードを調べたところ、Blog（Brave Search）・はてなブックマーク・RSSフィード
追従の3インポーターは既に「前回収集時と内容が同じならAIレビュー・DB書き込みをスキップする差分検知
（`findItemVersionByExternalUrl`）」を実装済みだったが、Qiita/Zenn/arXivの3つはこれがなく、毎日
全候補（arXivは既定`maxResults=50`）を無条件にAIレビューしていた。1件あたりOpenAI呼び出し1回＋
Supabase呼び出し6〜8回がかかるため、Qiita候補20件だけでも120〜180 subrequestsに達し、これが
subrequest上限超過とOpenAI課金の無駄打ちの両方の支配的な原因と判明した。

- `src/lib/importers/common.ts`: `findExistingItemVersions(externalUrls)`を新設。複数URLの
  既存versionを1回の`in`クエリでまとめて取得する（既存の`findItemVersionByExternalUrl`は1件ずつ
  問い合わせる版でblog/hatena/feedが使用、こちらは候補数に関わらず1クエリで済ませる）。
- `src/lib/importers/qiita.ts`・`arxiv.ts`: API一覧レスポンスの時点でURL・更新日時が揃っているため、
  `mapWithConcurrency`で回す前に`findExistingItemVersions`を1回呼び、変更なしの候補は
  `processImportItem`（AIレビュー＋upsert）を呼ばずに`reason: 'unchanged since last collection'`で
  即スキップするようにした。
- `src/lib/importers/zenn.ts`: Zennはslugだけでは記事URL・更新日時（`body_updated_at`）が分からず
  詳細取得（`fetchZennArticleDetail`）が必須なため、2段階に分割。①`fetchZennDetailSafely`で全slugの
  詳細を先に並行取得（失敗は従来通りskippedに吸収）、②取得できた記事URLをまとめて
  `findExistingItemVersions`で1回問い合わせ、③変更があった記事だけ`reviewAndUpsertZennArticle`
  （旧`processZennSlug`のAIレビュー＋upsert部分を切り出したもの）を呼ぶ。詳細取得自体（1候補1回の
  fetch）は避けられないが、変更なし記事へのAIレビュー・DB書き込みカスケードは丸ごと省略される。
- `wrangler.jsonc`・`src/worker.ts`（cron集約構成・ジョブ順序・Discord通知）は変更していない。
  Cloudflare Workers Paidプラン（$5/月、subrequest上限50→1000）へのアップグレードは、ユーザーが
  ダッシュボードで別途判断する事項としてスコープ外にした。

### 候補5（cronジョブ分割）の要否を実験で評価

候補1〜4実施後の実際のリクエスト量を、ローカルdevサーバー（ローカルSupabase接続）に対して
cronと同じ順序（フィード→Qiita→Zenn→ブログ→はてな→arXiv、リンク切れ検出は別途）で
各`POST /api/import/*`を実行し評価した。

- **フィード**: `requestsUsed: 1`、`fetched: 10, inserted: 10, skipped: 0`。未追従だった
  はてなブログの購読1件から過去記事10件が一度に「新規」と判定された（バックログのある
  フィードを初めて追従した実例）。新規1件あたり fetch(本文)1 + AIレビュー1 + ソースupsert
  1〜2 + アイテムupsert2（feedは`assumeNew`未適用のため既存行selectを含む）+ タグバッチ2
  （ジョブ全体で1回）を要するため、この1ジョブだけで概算60〜70 subrequestsに達した。
- **Qiita**: `fetched: 30, inserted: 1, skipped: 29`。既存29件は`already collected`で
  即スキップされ、ジョブ全体で概算10 subrequests程度。
- **Zenn**: `fetched: 48, inserted: 0`（全件スキップ）。詳細取得（1候補1fetch）自体が
  発生せず、ジョブ全体で概算5 subrequests程度。差分検知の効果が最も顕著。
- **arXiv**: `fetched: 20, inserted: 0`（全件スキップ）。概算5 subrequests程度。
- **リンク切れ検出**（`POST /api/import/check-links`）: `fetched: 100`
  （既定`LINK_CHECK_BATCH_LIMIT=100`の上限にヒット）。probeそのものは1件1fetchのため、
  新規記事の有無に関わらず概算100〜102 subrequestsが毎回固定で発生する。今回の
  バッチ化（候補1）はDB更新側だけを1回にまとめたものであり、probe fetch自体は削減対象外。
- **ブログ（Brave Search）**: 実行時に無関係な既存バグ（`URI too long`、後述）で失敗し
  今回は実測できなかったが、コード上`POKEMON_KEYWORDS`（6語）× 既定`pages=5`で
  最大30ページ分のBrave Search API呼び出しが発生する設計であり、これは発見（discovery）
  段階の呼び出しのため、既存記事のスキップ判定より前に発生し差分検知の削減対象外。
  概算30〜35 subrequests程度が新規/既存に関わらず固定でかかる。
- **はてなブックマーク**: `HATENA_CRAWL_DELAY_MS=5000`による意図的な遅延のため今回は
  完走を待たなかったが、キーワード6語分のRSS取得（概算6 subrequests）が発見段階の
  固定コストとして発生する。

#### 結論

候補1〜4は「既知記事1件あたりの追加コスト」をほぼ0まで削減する効果は確認できた
（Zennの詳細取得省略、Qiita/arXivの即時スキップなど）。しかし、**新規/既存の判定に
関わらず毎回固定で発生する2つの大きなコスト**——リンク切れ検出のprobe fetch（既定
100件）と、ブログ（Brave Search）の発見段階のページネーション（6語×5ページ＝最大30件）
——が未対応のまま残っており、この2つだけで概算130 subrequents前後に達する。
これに他5ジョブの固定コスト（各5〜15程度）を足すと、新規記事が全く無い「静かな日」でも
合計は150〜170 subrequests程度になり、Cloudflare無料/標準プランの上限（50/呼び出し）を
大きく超える。今回フィードで観測したような新規記事のバックログが発生した日は、
1ジョブだけで上限を超えることもある。

**したがって、候補5（cronジョブの複数呼び出しへの分割）は依然として必要と判断する。**
`wrangler.jsonc`の`triggers.crons`は現在2件（週次レビュー・日次収集）しか登録しておらず、
アカウント上限5件のうち3件の空きがあるため、ワイルドカードcron等の追加の仕組みを
導入しなくても、日次収集を2〜3個の独立したCron Triggerエントリ（例: リンク切れ検出を
専用の発火時刻に分離、収集6ジョブを2グループに分割）に分けるだけで対応できる見込み。
具体的な分割方針は別途ユーザーと合意のうえで実装する。

### リンク切れ検出: 追跡を諦めるか検討し、バッチサイズの自動算出＋専用cron化を実施

ユーザーから「リンク切れ検出は諦めて追跡しないのはどうか」との提案があったため検討した。
`link_status='broken'`は一覧・検索・RSS・サイトマップから記事を隠す実用機能
（`src/lib/catalog.ts`の`query.neq('link_status', 'broken')`）であり、追跡をやめると
削除済み・非公開化された元記事へのリンクが一覧に残り続けることになる。記事内容の変更追跡
（内部の再インデックス精度の話）とは異なり外部向けのコンテンツ品質に直接影響するため、
「追跡を諦める」のではなく「1回のチェック件数を絞って残す」方針を提案し、了承を得た。

さらに「記事数が増えたら頻度を上げる必要があるか」という質問への回答として、頻度固定・
バッチサイズだけを対象総数から自動算出する設計を提案し、実装した。

- **`src/lib/importers/link-check.ts`**: `LINK_CHECK_BATCH_LIMIT`が未指定の場合、
  `resolveAdaptiveBatchLimit()`が対象件数（`ai_accepted=true`かつ`external_url`がある件数）を
  1回のcountクエリで取得し、`ceil(対象件数 / recheckIntervalDays) + 余裕分(5)`を算出、
  `SAFE_MAX_BATCH_LIMIT=40`（probe fetch以外の固定コストを差し引いた、1回の呼び出し単体で
  subrequest上限に収まる安全上限）で頭打ちにする。件数が増えても手動調整が要らず、
  上限に達した後は「一巡にかかる日数が伸びる」形で緩やかに劣化するだけで、
  subrequest超過には戻らない。`LINK_CHECK_BATCH_LIMIT`を明示指定した場合はそちらを優先する
  （動作確認・緊急対応用の上書きは維持）。
- **`src/worker.ts`・`wrangler.jsonc`**: リンク切れ検出は「新着」の概念がなくprobe fetch
  （1件1fetch）が差分検知で削減できないため、日次収集6ジョブとは別に専用のCron Trigger
  （`"30 0 * * *"`＝JST 09:30、`LINK_CHECK_CRON`）に分離した。アカウントのCron Trigger上限
  （5件）のうち3件が空いていたため、追加の仕組み（ワイルドカードcron等）なしで対応できた。
  `runDailyJobsSequentially`内の「ブログの後にリンク切れ検出を呼ぶ」処理は削除した。
- **`.env.example`**: `LINK_CHECK_BATCH_LIMIT=100`をコメントアウトし、既定は自動算出である旨を
  注記した。

#### 検証（リンク切れ検出の自動バッチサイズ・専用cron化）
- `npm run build`・`npm test`（既存103件）green。
- ローカルSupabase（`ai_accepted=true`かつ`external_url`がある対象件数330件）に対し
  `POST /api/import/check-links`を実行し、`fetched: 40`（`ceil(330/7)+5=53`が
  `SAFE_MAX_BATCH_LIMIT=40`で頭打ちになった値と一致）を確認した。
- `GET /api/import/check-links`で`batchLimit`が既定値として返らなくなり（自動算出のため
  固定値を持たない）、`concurrency`/`recheckIntervalDays`/`timeoutMs`は従来通り返ることを確認した。

### 新着記事・論文の急増日に備えた「1回あたりの新規処理件数」上限を全6収集ジョブに追加

ユーザーから「新着記事・論文が急増する日があっても問題なく動作し続けられるか」との確認依頼が
あったため検証した。既存の差分検知（既知記事のスキップ）は「1回の実行で新規として処理する
件数」自体には上限がなく、既存URL判定を通過した新規候補は件数に関わらず無条件に全件処理される
実装だったことが判明した。

具体的な悪化ケースを試算すると、arXiv（既定`maxResults=20`）は全件新規なら1ジョブ単独で
約65 subrequests、Zenn（`pages=1`で実測48件/回）は全件新規なら約190〜200 subrequestsとなり、
いずれも単独でCloudflareの上限（50/呼び出し）を超えうることを確認した。フィードは購読数が
`blog`/`hatena`の記事採用時に自動登録され続けるため、購読数の増加に伴い構造的に悪化する
リスクもある。さらに、一度どこかのジョブが上限を超えると、`sendOperationalAlert`自体も
subrequestを使うため同一呼び出し内の以降のジョブが道連れで失敗し、記録も通知もされない
（2026-07-09発覚の当初の障害と同じ挙動）ことも確認した。

対応として、リンク切れ検出の`resolveAdaptiveBatchLimit`と同じ考え方で、6ジョブ全てに
「1回の実行でAIレビュー・DB書き込みを行う新規件数の上限（`maxNewItemsPerRun`）」を追加した。
上限は`.env`で管理する（ユーザー指定）。超過分は本文取得・詳細取得自体を省略してスキップし
（`reason: 'exceeded max new items per run'`）、既存URL判定には残るため次回実行時に自然に
再度候補となる（記事・論文が失われるわけではない）。

- **`qiita.ts`/`arxiv.ts`**: 既存URL判定後の新規リストを構築し、`maxNewItemsPerRun`件までの
  URLだけを処理対象にする`Set`を作って判定を差し替えた（既定: Qiita 10、arXiv 10）。
  `QIITA_MAX_NEW_PER_RUN`/`ARXIV_MAX_NEW_PER_RUN`で上書き可能。
- **`zenn.ts`**: 既に構築済みの`newArticles`を`maxNewItemsPerRun`（既定8、詳細取得が1件分
  多いため他より控えめ）で`slice`し、超過分は**詳細取得自体を行わずに**スキップする。
  `ZENN_MAX_NEW_PER_RUN`で上書き可能。
- **`blog.ts`/`hatena.ts`**: 既に構築済みの`newCandidates`を`maxNewItemsPerRun`（既定6、
  発見段階のBrave Search呼び出し自体が固定コストのため控えめ）で`slice`し、超過分は
  本文取得自体を行わずにスキップする。`BLOG_MAX_NEW_PER_RUN`/`HATENA_MAX_NEW_PER_RUN`で上書き可能。
- **`feed.ts`**: `discoverCandidates`が返す候補（既に新規のみ）を`maxNewItemsPerRun`
  （既定10）で`slice`する。`FEED_MAX_NEW_PER_RUN`で上書き可能。
- **`.env.example`**: 全6ジョブ分の`*_MAX_NEW_PER_RUN`をコメントアウト状態で追記した
  （既定値のままでよいため、明示的に上書きしたい場合のみ設定）。

#### 検証
- `npm run build`・`npm test`（既存103件）green。
- ローカルSupabaseに対し`POST /api/import/qiita`（`perPage: 30, maxNewItemsPerRun: 1`）を
  実行し、新規9件中1件だけがAIレビュー対象になり（棄却）、残り8件が
  `reason: 'exceeded max new items per run'`でスキップされることを確認した。
- 同様に`POST /api/import/zenn`（`pages: 2, maxNewItemsPerRun: 1`）を実行し、新規3件中1件が
  採用（`inserted`）、残り2件が`reason: 'exceeded max new items per run'`でスキップされる
  ことを確認した（詳細取得前に上限判定されるため、超過分の詳細取得subrequestも発生しない）。

### ブログ（Brave Search）専用cronへの分離・検索キーワードの巡回実行

ユーザーから「`maxNewItemsPerRun`は単一cronで全ジョブが同時に最大値になるワーストケースを
想定しているか」と問われ、試算し直した結果、**Blogジョブ単体**でも発見段階（Brave Search、
6キーワード×既定`pages`）の固定コストだけで単独で約54 subrequestsに達し、他ジョブと
合算する前の時点で単独でも上限（50/呼び出し）を超えうることが判明した。`maxNewItemsPerRun`は
新着記事の処理件数（AIレビュー・DB書き込み）は抑えるが、発見段階の固定コストまでは
削減できないため、6ジョブ合計のワーストケースは依然として抑えられていなかった。

続けてユーザーから「日次の単一cronジョブを、ソースを切り替えながらソース数だけ走らせたら
どうか」との提案があり、Cloudflareのcron trigger上限（アカウント全体5件）は「登録する
パターン文字列の数」であって「1日の発火回数」ではないため、`0 1,5,9,13,17,21 * * *`の
ように時刻をカンマ区切りにした1エントリで1日6回発火できることを確認した（過去にPR #24で
実際に使っていた「ワイルドカードcron + `scheduledTime`振り分け」と同じ手法）。

さらにユーザーから「Brave Searchの検索語（`POKEMON_KEYWORDS`）は今後も変わる可能性が高い」
という指摘があり、単純に「1日6回＝6ジョブ分岐」とハードコードするのではなく、**発火時刻から
求めた通し番号をその時点のキーワード数で割った余り**でキーワードを1つ選ぶ方式にした。
これによりキーワード配列が増減しても、cron設定・コードのどちらも変更不要になる（増えれば
一巡に要する日数が伸びるだけ）。

実装:
- `wrangler.jsonc`: `crons`に`"0 1,5,9,13,17,21 * * *"`を追加（`BLOG_CRON`、4件目、
  アカウント上限5件のうち1件の空きを残す）。
- `src/worker.ts`: `DAILY_COLLECTION_JOBS`からブログを除外し、`BLOG_CRON`専用の分岐で
  `runScheduledBlogImport(controller.scheduledTime)`を呼ぶように変更。同関数内で
  `POKEMON_KEYWORDS[resolveBlogKeywordIndex(scheduledTime, BLOG_ROTATION_SLOT_MS, POKEMON_KEYWORDS.length)]`
  を`query`として`syncBlogCollection`に渡す。
- `src/lib/importers/blog-rotation.ts`（新規）: `resolveBlogKeywordIndex`を独立した
  純粋関数として切り出した。blog.ts本体はBrave/Supabase等の外部依存を持つため`node --test`で
  直接importできず（既存の`process-import-item.ts`・`link-status.ts`と同じ理由・同じ方針）、
  cloudflare:workers等に依存しないファイルに分離してユニットテスト可能にした。
- `tests/blog-keyword-rotation.test.ts`（新規）: 発火間隔ぶん時刻が進むと通し番号が
  ちょうど1つ進むこと、キーワード数で一巡すると先頭に戻ること、キーワード数が変わっても
  巡回範囲が追従することを確認するテストを追加。

#### 検証
- `npm run build`・`npm test`（106件、新規3件含む）green。
- ローカルSupabaseに対し`POST /api/import/blog`へJSONボディ`{"query":"pokeapi","pages":1,"count":5}`
  を渡して実行し、`queries: ["pokeapi"]`・`requestsUsed: 1`となることを確認した
  （cronが実際に呼ぶのと同じ「単一キーワードだけをqueryに渡す」経路の動作確認）。
- この確認の過程で、`query`を渡さず全キーワード既定実行（`.env`の`BLOG_PAGES=15`）を
  誤って呼び出した際に、既知の`URI too long`エラー（本件とは無関係、2026-07-09に発見済みの
  既存バグ、未対応）が改めて再現することを確認した。今回の変更で1回あたりの検索候補
  ボリュームが減るため再現頻度は下がる見込みだが、根本修正は引き続きスコープ外。

#### `MAX_NEW_PER_RUN`の再調整要否を検討（結論: 変更なし）

ブログの専用cron分離を受け、ユーザーから「`MAX_NEW_PER_RUN`を再調整」との依頼があったため、
全ジョブのワーストケースを再計算した。

- **日次バンドル5ジョブ（Qiita/Zenn/はてな/Feed/arXiv）**: Blogが抜けたことで共有invocationの
  予算に余裕ができたように見えるが、残る5ジョブだけでもワーストケース合計は約164
  subrequests（Qiita~25 + arXiv~25 + Zenn~29 + はてな~40 + Feed~45）に達し、Blog分を
  差し引いても既に50を大きく超える。バンドル自体の分割（候補5の残課題）が未解決なため、
  これら5ジョブの上限を上げる余地はそもそも無く、**変更しない**。
- **Blog**: 1呼び出しあたりの固定コストを、`.env`の実際の値（`BLOG_PAGES=15`、コード既定の
  5ではない）で計算し直すと、1キーワード分でもBrave呼び出し最大15＋既存判定・タグバッチ・
  実行記録で固定約20〜23。新規記事1件あたりのコストはfetch＋レビュー＋source upsert＋
  item upsert（`assumeNew`未使用のためselect＋upsert）で約5〜6。この前提だと安全に収まる
  上限は4〜6程度で、現状の`BLOG_MAX_NEW_PER_RUN=6`は既にこのレンジの上限寄りであり、
  明確に引き上げられる根拠は薄いことが判明した。6キーワード同時実行（旧・最大90）と比べれば
  1呼び出しあたりの固定コストは6分の1に下がっており「安全になった」のは確かだが、それは
  「絶対量が減った」効果であって「新規処理枠を増やせる」ほどの余裕ではない。
- ユーザーに実際の数値を提示のうえ確認し、**`BLOG_MAX_NEW_PER_RUN`は6のまま変更しない**方針を
  確定した。上限を上げたい場合は先に固定コストの支配項である`BLOG_PAGES`（現状15）を
  下げる必要があることも共有済み。コード・`.env`・`.env.example`はいずれも変更していない。

## itemsテーブルへの収集経路（collection_route）記録（`feat/items-collection-route`）

事後解析のため、各記事を実際にどの収集ジョブが取り込んだかを記録したいとの要望があった。
具体例として「Qiita記事のはずがBrave Search経由でしか見つかっていない」といったケースを
検出できれば、当該インポーターの検索方法（クエリ・ページ数等）の改善余地が分かる、という
狙い。

- `sources.type`は掲載元プラットフォーム単位（Blog(Brave Search)・はてなブックマーク・RSS
  フィード追従の3ジョブはいずれも`'blog'`）でしか区別できず、実際の発見方法（経路）は
  `items.metadata->'provenance'->>'source'`にしか記録されていなかった（JSONB内でインデックス
  も無く、事後解析のたびにパスを掘る必要があった）。この値は全7インポーター
  （qiita/zenn/arxiv/note/blog/hatena/feed）が挿入時から一貫して設定していたため、専用列へ
  昇格しバックフィルする方針にした。
- `migrations/024_add_items_collection_route.sql`（新規）: `items.collection_route text`列を
  追加し、既存行は`metadata->'provenance'->>'source'`から一括バックフィル。絞り込み・集計用に
  インデックスも追加。
- `src/lib/importers/common.ts`: `ItemUpsertPayload`に`collectionRoute: string`を追加し、
  `upsertItemByExternalUrl`のupsert対象に`collection_route: payload.collectionRoute`を渡す。
- 全7インポーターの`upsertItemByExternalUrl`呼び出しに`collectionRoute`を追加。値は既存の
  `metadata.provenance.source`とそれぞれ一致させた：`qiita-importer`/`zenn-importer`/
  `arxiv-importer`/`note-importer`/`brave-search-importer`/`hatena-bookmark-importer`/
  `feed-importer`。

### 検証
- `npm test`（103件）・`npm run build` green。
- ローカルSupabaseへ`node scripts/db/run-migrations.mjs`でマイグレーション適用し、既存506件を

### 今後の分析例
`collection_route`と`sources`（ドメイン・プラットフォーム）を突き合わせることで、例えば
「本来Qiita APIで見つかるべき記事が実際には`brave-search-importer`/
`hatena-bookmark-importer`経由でしか収集されていない」ようなケースを`sources`結合や
ドメイン（qiita.com等）とのクロス集計で検出できる。ただし現状`blog.ts`/`hatena.ts`は
`EXCLUDED_BLOG_DOMAINS`でqiita.com等を検索結果から除外しているため、この具体例が実際に
発生するのは除外ロジックの不備がある場合に限られる。より一般的には「特定ジョブの検索語・
ページ数では拾えていないが、他ジョブ経由では見つかっている技術的に近い記事」のパターン発見に
使う想定。

### 発火時刻を一時変更しての実動作確認 → 失敗、方法に不備

過去の前例（PR #36/#39）に倣い、`wrangler.jsonc`のcron時刻を数分後に変更してCloudflare上で
実際に発火するかを2回試行したが（05:12-14 UTC、05:33-35 UTC）、`wrangler tail`・
`GET /api/import/runs`のいずれにも発火の痕跡が見られなかった。調査の結果、テスト方法自体に
不備があったと判明した：`wrangler.jsonc`のcron文字列だけを変更しても、`src/worker.ts`の
`scheduled()`ハンドラは`DAILY_CRON`/`LINK_CHECK_CRON`/`BLOG_CRON`という**コード側の定数**と
`controller.cron`を比較してジョブを振り分けており、この定数を合わせて変更していなかった。
一時変更はPR #44で元のスケジュールへ復旧した。この手法（cron trigger登録のみの変更）は
今後使う場合、`src/worker.ts`側の判定定数も同時に変更する必要がある。
