# 2026-07-10

## 本番DBの偽陽性記事（「ポケモンに学ぶシステム開発」シリーズ）を非公開化

朝のDiscord日次まとめ通知に、[filter-accuracy.md](../optimization/filter-accuracy.md) で偽陽性の典型例として分析済みの「ポケモンに学ぶシステム開発」シリーズ記事が多く含まれているとの指摘を受け、本番DBを確認した。

### 確認結果

- id=612, 613, 615, 617, 343（同シリーズ）が本番で `ai_accepted=true` のまま残存していた。
- 原因: STEP形式プロンプト修正（PR #56、2026-07-09 21:07 JST マージ・デプロイ）より前に収集された記事のため、旧プロンプトの判定がそのまま残っていた。`retag-existing-items.mjs` の棄却分岐は「既に公開済みのアイテムを自動で非公開・削除にはしない」設計のため、filter-accuracy.md記載の「本番反映」作業（同5件の`--dry-run`確認）でも実際の `ai_accepted` は書き換わっていなかった。
- 加えて id=634（同シリーズの新規話「第1章：戦術編 第2話」、07-09 17:06 JST収集）がPR #56デプロイ**直前**に収集され、旧プロンプトのまま `ai_accepted=true` で混入していることも判明。こちらは新プロンプトでの再判定が未検証のため今回は対象外。

### 対応

filter-accuracy.mdの実験で新プロンプト（STEP形式）による棄却判定が確認済みの5件（id=343, 612, 613, 615, 617）について、追加のOpenAI呼び出しをせず、本番DBの `ai_accepted` を直接 `false` に更新した（使い捨ての読み取り/書き込みスクリプトで実施、実行後に削除）。`items.ai_accepted=false` は `catalog.ts` の一覧・検索クエリで除外されるため、これで一覧・検索・日次通知の対象から外れる（詳細ページ・ブックマーク一覧からは引き続き参照可能、summary/tagsは変更していない）。

### 残課題（当初）

- id=634（新規話、未検証）: 新プロンプトでの再判定が必要。
- id=61, 70, 138, 613 の判定不安定性（filter-accuracy.md「残課題」参照）は未解消のまま。
- 今回の5件を除く残り記事全体への `retag-existing-items.mjs` 適用の要否・タイミングは引き続き未決定（OpenAI課金を伴うため実行前にユーザー確認が必要）。

### 追加で浮上した課題: reason/confidenceの追検証性

`shouldPreserveAcceptedItem`（`src/lib/importers/process-import-item.ts:30`）により、一度 `ai_accepted=true` になった記事は再収集のたびに新しい判定が出ても書き込みごとスキップされるため、`metadata.ai.reason/confidence` が最初に採用された時点のまま凍結され、プロンプトを直しても更新されない。id=5のreasonが旧プロンプト形式のまま止まっていたのはこれが原因で、「今のプロンプトなら実際どう判定されるか」をDBだけから知る手段がない。

改善案（未実装、ユーザー判断待ち）: `ai_accepted`（公開可否フラグ、自動格下げしない現行方針は維持）とは別に、直近の再レビュー結果（reviewed_at/accepted/reason/confidence/prompt_version）を常に上書き記録する仕組みを追加し、`buildSystemPrompt()`のハッシュ等を `prompt_version` として記録することで「現行プロンプトと異なるバージョンで判定された記事」をSQLで即座に抽出できるようにする。ユーザーからは「まず今日見つかった偽陽性への対応を優先」の指示があったため、スキーマ改善は未着手。

### 事前調査で判明した計画時点との差分

- OpenAlex APIは無料キー必須に変更されていた（計画時点は「無料枠あり」の想定）。ユーザーが
  `openalex.org/settings/api` で取得したキーを `.env`（main/worktree双方）に設定済み。無料枠は
  1日$1相当（フィルタ検索1万コール/検索1000コール等）で本サイトの規模には十分。
- アブストラクトは平文で提供されず、`abstract_inverted_index`（逆インデックス）形式のみ
  （OpenAlex側は出版社の権利関係を理由と明記）。arXiv側の未確認事項（アブストラクト転載可否）
  と同種の論点として `docs/plan/paper.md` に記録した。
- arXivとの重複懸念には、OpenAlexのWork objectがarXivのアブストラクトページURLを含む場合に
  arxiv.ts と同じ正規化URLを `external_url` として使う設計で対応（`items.external_url` の
  UNIQUE制約を介して自然に同一行へ収斂させる。新規の重複排除ロジックは追加しない）。

## OpenAlexのmaxNewItemsPerRun実験・cron統合（`feat/openalex-cron-tuning`）

本番反映後、ユーザーから「既定の`maxNewItemsPerRun=10`は妥当か」との質問を受けた。
再検証の結果、10はarXivの値をそのまま踏襲しただけで、OpenAlex向けに実測したものではなく、
またOpenAlexはcron未統合（手動起動のみ）のため、arXivのように「今日処理しきれなかった分は
明日のcronが自動で拾う」という前提が成り立たないことを指摘した。ユーザーからは
「実験で検証し、確定したらcronの日次ジョブに追加」の指示を受け、以下を実施した。

- `scripts/eval/eval-subrequests.mjs`のROUTESにopenalexを追加。
- ローカルdev serverを`DEBUG_SUBREQUEST_COUNT=1`（`.dev.vars`、検証後に削除済み）で起動し、
  既存の実測手法（固定コスト計測→対象を1件削除→`maxNewItemsPerRun=1`で新規1件あたりのコスト計測）
  でopenalexルートを実測した。固定コスト5・新規1件あたり3という結果を得た
  （詳細・既定値決定の根拠は[docs/issue/cron-subrequest-limit.md](../issue/cron-subrequest-limit.md)
  「2026-07-10: OpenAlexインポーター追加に伴う実測・cron統合」を参照）。
- 新規1件あたりが当初のarXivからの類推値「2」ではなく「3」だった理由として、
  `ai_prompt_hashes`テーブル（PR #65）追加に伴う`recordPromptHashIfNew`のupsert呼び出しを特定した。
  この関数は全インポーター共通のため、cron-subrequest-limit.md記載のqiita/zenn/arxiv/feedの
  既存の実測値（PR #65より前のもの）も同様に過小評価されている可能性があると判明したが、
  再測定は本対応のスコープ外として持ち越した。
- 既定値を10→12へ引き上げ（ワーストケース43、qiita/zenn/arxivと同程度の安全マージン）。
- `src/worker.ts`の`DAILY_SLOT_JOBS`に分30のスロットとして追加し、`wrangler.jsonc`の
  `DAILY_CRON`文字列にも`30`を追加した（新規のCloudflare Cron Trigger登録は不要）。
  実行順序はarXiv・リンク切れ検出の後、日次まとめ通知の前とし、OpenAlexのarXiv重複回避
  （arXiv importerと同じ正規化URLを使う設計）がその日のarXiv収集結果に対しても効くようにした。
- `npm run build`・`node --test`で確認済み。

作業は`feat/openalex-cron-tuning`ブランチで実施（PRは別途作成、マージはユーザー確認後）。

## 本番DBの重複item統合

ユーザーから「本番DBの重複している論文を統合して」と依頼を受けた。既存の`scripts/db/detect-duplicate-items.mjs`（読み取り専用）を本番DBに対して実行したところ26組の候補が見つかったが、`kind='paper'`同士の組と`kind='article'`（Qiita/Zenn等ブログ）同士の組が混在していたため、`id`を指定して`kind`/`source_id`/`ai_accepted`を個別確認し、以下のように分類した。
