## Cron Trigger登録数上限により、はてな収集・週次レビューが一度も自動実行されていなかった問題の修正（PR #24）

「本番環境での記事の定期収集は成功しているか」との確認依頼を受け調査した。

### 調査経緯
- `GET /api/import/runs`（`import_runs`テーブル）を確認したところ、全期間で1行（2026-07-06 22:00のnote失敗）のみで、Qiita/Zenn/Blogの実行履歴が皆無だった。
- Cloudflare GraphQL Analytics API（`workersInvocationsScheduled`データセット）で実際のcron発火履歴を直接確認したところ、Qiita/Zenn/note/Blog/リンクチェックは`success`で正常に発火し続けており、本番`items`にも新規記事が実際に増え続けていることを確認。**収集自体は動いていた。**
- 一方、Cloudflare API（`GET .../workers/scripts/poke-research/schedules`）で本番に実際に登録されているcronを直接確認すると、`wrangler.jsonc`で定義した7件のうち5件（Qiita/Zenn/note/Blog/リンクチェック）しか登録されておらず、**はてなブックマーク収集と週次DBレビューの2件が欠落**していた。
- 原因を切り分けるため`wrangler triggers deploy`を試したところ失敗し、Cloudflare APIへ直接PUTしてエラー本文を取得したところ `code 10072: exceeded the limit of 5 cron triggers` と判明。**現行のCloudflareアカウントプランはCron Trigger登録数がアカウント全体で5件までという上限があり**、はてな・週次レビューを追加した時点（過去のいずれかのデプロイ）で登録がサイレントに失敗し続けていたことが根本原因だった。
- `import_runs`がほぼ空である件（Qiita/Zenn/Blogの実行履歴が記録されていない件）は、cron自体は動いているため上記とは別の問題として原因未特定のまま残っている。次回調査が必要。

### 対応
ユーザーに選択肢を提示し、「日次収集ジョブ群を1本のcronに統合する」案を採用。`fix/cron-trigger-limit`ブランチで実装：
- `wrangler.jsonc`: 日次6ジョブ（Qiita/Zenn/note/Blog/リンクチェック/はてな）を個別の6エントリから `"*/30 21-23 * * *"`（21:00〜23:30 UTCを30分刻みで1日6回起動）の1エントリへ集約。週次レビュー（`30 20 * * 1`）は据え置き。登録エントリ数は7→2に削減。
- `src/worker.ts`: 集約後は`controller.cron`が6スロットすべてで同一文字列になるため、`controller.scheduledTime`（UTC時刻）から時:分を算出してジョブを振り分ける方式に変更（`DAILY_SLOT_HANDLERS`）。各ジョブの起動時刻・実行順序は変更なし。
- `npm run build`・`npm test`（88件）でCI green確認後、PR #24として通常マージ。マージ後、本番へ`npx wrangler triggers deploy`を実行し、Cloudflare側の登録cronが意図通り2件（`30 20 * * 1` / `*/30 21-23 * * *`）になったことをAPIで直接確認した。

### 残課題
- `import_runs`にQiita/Zenn/Blogの実行履歴が記録されない原因は未調査のまま。次回の日次スロット到達後、`GET /api/import/runs`で記録されるか確認すること。
- はてなブックマーク収集・週次DBレビューは、今回の変更で初めて自動実行の対象になる（過去は一度も自動実行されていなかった）。次回の23:30 UTC（はてな）・次の月曜20:30 UTC（週次レビュー）で実際に発火し完了することを確認するとよい。

## noteをBrave Search経由のブログ収集に切り替える検証（`feat/note-via-blog-search`）

上記403の代替案として、noteの専用API（`/api/v3/`）を経由せず、他の個人ブログ同様Brave Search経由で発見しHTML抽出で収集できないか検証した。

### 検証内容
- `src/lib/importers/keywords.ts`の`EXCLUDED_BLOG_DOMAINS`からnote.comを外し、`KNOWN_BLOG_PLATFORMS`に`{ domain: 'note.com', name: 'note' }`を追加（note.tsと同じsource originUrlに集約されるように）
- `npx wrangler versions upload`で本番トラフィックを切り替えずにプレビューバージョンを作成し、そのプレビューURLへ`POST /api/import/blog`を手動実行して本番Cloudflare Workers環境から実際にnote.comへ到達できるか確認した
- **結果**: note.comのトップページ・実記事URL（`https://note.com/tukudaniptcg/n/nadfde04e2165`）とも200 OKで正常に取得・本文抽出・AIレビューまで到達した（AIレビュー結果は主題不一致でskippedだったが、fetch自体は成功）。**403ブロックは`/api/v3/`エンドポイント固有で、Cloudflare Workersの送信元IP自体が広くブロックされているわけではない**ことが確認できた
- 検証中、記事タイトルをそのままクエリにしたところ、Brave検索で自サイト（poke-research.com、その記事の要約を掲載済み）がヒットして誤ってitemとして挿入される事故が発生（id=441、削除済み）。クエリの選び方によっては自己参照が起きうる点は留意
