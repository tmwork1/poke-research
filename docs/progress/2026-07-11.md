# 2026-07-11

## ブックマーク失敗の原因調査・修正（`fix/bookmark-user-sync`、worktree）

ユーザーから「記事のブックマークに失敗する」と報告を受け、調査した。

### 調査

- ローカル開発環境（`astro dev`のダミーユーザー、GRANT・RLS状態）ではDB直接操作・APIとも正常に動作し、再現しなかった。
- ユーザーに発生環境を確認したところ本番（`poke-research.com`）で発生。さらに「`tmtm.holmes@gmail.com`は失敗するが`tmtm.pcmail@gmail.com`は成功する」との追加情報を得た。
- 本番DBを読み取り専用で確認したところ、`auth.users`には3ユーザーとも存在するが、`public.users`には`tmtm.holmes@gmail.com`（id=`79a1cdb5-32bb-441c-a6c7-e45a8f32cb84`）のみ行が無いことが判明した。
- `bookmarks.user_id`は`public.users(id)`への外部キー制約（`migrations/007`）を持つため、`public.users`に行が無いユーザーはブックマーク追加時にFK違反（`23503`）となる。`src/lib/bookmarks.ts:11-13`がこのエラーコードを一律「item not found」に丸めるため、実際の原因（ユーザー未登録）が隠れていた。
- `public.users`への同期は`migrations/006_link_users_to_auth.sql`の`on_auth_user_created`トリガー（`AFTER INSERT ON auth.users`）のみに依存しており、既存行には遡って発火しない。holmesアカウントの`auth.users`作成（07-05 11:17:23）がpcmailアカウント（07-05 11:42:35）より先で、この間にmigration 006が適用されたため、holmesのみ同期対象から漏れたとみられる。

### 対応

`src/pages/api/auth/callback.ts`で、`exchangeCodeForSession`成功後に`public.users`へのupsert（`onConflict: 'id', ignoreDuplicates: true`）をアプリ側でも行うよう修正した。トリガーに依存せず、ログインのたびに同期漏れを自己修復する。

本番DBの`tmtm.holmes@gmail.com`行を手動追加する案もユーザーに確認したが、「実行しない」との判断のため見送り、次回ログイン時に上記修正で自動修復されるのを待つ方針とした。

### 検証

- `npm run build`（成功）。
- `node --test "tests/**/*.test.ts"`（129件全通過、既存回帰なし）。
- ローカルDBで同一のupsertロジック（`onConflict: 'id', ignoreDuplicates: true`）を直接実行し、既存ユーザーへの冪等upsertがエラーなく成功すること、`auth.users`に対応行が無いIDではFK違反になることを確認した（実運用では`data.user.id`は必ず`auth.users`に存在する値のため後者は発生しない）。
