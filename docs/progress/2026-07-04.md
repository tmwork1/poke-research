- [docs/review/2026-07-04.md](../review/2026-07-04.md) のレビュー指摘に沿って、以下を修正した。
  - Qiita インポートを記事単位の try/catch に変え、1 件の失敗（AI レビュー/DB 書き込みエラー）でバッチ全体が止まらないようにした。あわせて記事処理を limit 付き並列実行にして収集全体の待ち時間を短縮した。
  - `sources.origin_url` / `items.external_url` に UNIQUE 制約を追加する `migrations/002_add_unique_constraints.sql` を作成・適用し、Qiita インポートの select→insert/update を `upsert(onConflict)` に置き換えて TOCTOU による重複行の可能性をなくした。
  - `item_tags` の同期を全削除→再作成から、既存タグとの差分だけ insert/delete する方式に変え、途中失敗時にタグが失われたまま残らないようにした。
  - `items`/`search` ページで `sourceId` の数値パースを検証するようにし、不正な値で SSR が 500 にならないようにした。あわせて `src/lib/params.ts` に数値パースの共通ヘルパーを切り出し、API ルート・Qiita インポーター・Astro ページの重複実装を統一した。
  - `/api/items/[id]` の PUT/PATCH が更新後に `fetchCatalogItemById` で読み直すようにし、GET と同じレスポンス形状（source/tags/annotations 込み）に揃えた。
  - トップページの `fetchCatalogOverview` 呼び出しを try/catch し、DB 取得に失敗してもページ全体が 500 にならず、フォールバック表示に留まるようにした。
  - `src/lib/supabase.ts` のモジュールスコープにあった重複の env 未設定警告を削除した（`getSupabaseClient()` 内のチェックで足りるため）。
  - レビュー内の「Qiita 検索条件チューニング用の dryRun オプション」は将来提案として見送り、今回は対応していない。
- 上記レビュー修正一式を検証（`npm run build` / `scripts/test-db.mjs`）のうえコミットした。
- M3 の残タスクのうち「収集ジョブの定期実行＋失敗時の再実行手順」に着手し、Cloudflare Cron Triggers 方式を採用した（GitHub Actions 案・拡張先行案はユーザー確認のうえ見送り）。
  - API ルートの `readDefault*` を `src/lib/importers/qiita.ts` の `resolveQiitaSyncOptions` に集約し、手動起動（`POST /api/import/qiita`）と定期実行の両方で同じ既定値解決ロジックを使うようにした。
  - `src/worker.ts` を新設し、`@astrojs/cloudflare/handler` の `handle` で fetch をそのまま委譲しつつ、`scheduled` ハンドラで `syncQiitaCollection` を起動するようにした。`wrangler.jsonc` の `main` をこのファイルに向け、`triggers.crons` に `0 18 * * *`（毎日 JST 翌 3:00）を追加した。
  - `wrangler dev --test-scheduled` で `/cdn-cgi/handler/scheduled` を手動呼び出しし、実際に Qiita 収集が動く（fetched 20 / inserted 18 / skipped 2）ことと、通常の HTTP ルート（`/` `/api/items`）が引き続き 200 で応答することを確認した。
  - 失敗時の再実行手順は、記事単位の失敗は既存の try/catch で `skipped` に吸収される前提のもと、ジョブ全体が落ちた場合のみ `POST /api/import/qiita`（または次回 cron）を再実行すれば冪等に復旧できる、という運用を README に明記した。
- M3 の残タスク「Zenn / GitHub / RSS / note への収集対象拡張」に着手し、Zenn を選択した（GitHub は公式 API で安定、note/Zenn は非公式 API という違いをユーザーに提示のうえ、Zenn からユーザーが選択）。
  - Qiita と Zenn の 2 ソース目を作る過程で、DB 書き込み・並列実行・記事単位の try/catch といった共通部分を `src/lib/importers/common.ts` に切り出した（`mapWithConcurrency` / `stripHtml` / `ensureTags` / `syncItemTags` / `upsertSourceByOriginUrl` / `upsertItemByExternalUrl` / `processImportItem`）。`qiita.ts` もこの共通モジュールを使うよう refactor し、既存の Qiita 取り込み動作に変化がないことを `wrangler dev` 経由の実取り込み（fetched 3 / inserted 2 / updated 1）で確認した。
  - Zenn は公式ドキュメントの無い API のため、`zenn.dev/api/articles?topicname=` と `zenn.dev/api/articles/{slug}` を直接検証（curl）して実際のレスポンス形を確認した上で実装した。検索はキーワードではなく `topic`（既定 `pokemon`）で絞り込む（Zenn 側にキーワード全文検索 API が無いため）。
  - 実装時、`/api/articles/{slug}` のレスポンスが `{ article: {...} }` で本体を包んでいることに気づかず最初の実装で全件 skipped になったが、curl で実レスポンスを確認して修正した（WebFetch 経由の要約では実際のネスト構造を見誤ったため、外部 API のレスポンス形は必ず curl などで生データを確認する必要がある、という教訓）。
  - `src/worker.ts` の `scheduled` ハンドラを、`controller.cron` の値で Qiita（`0 18 * * *`）/Zenn（`30 18 * * *`）どちらのジョブを起動するか振り分けるように拡張した。
  - `wrangler dev` 経由で実際に Zenn 収集（topic=pokemon, 1ページ=48件）を実行し、AI レビューが ROM 改造・チート手順など違法性の高い記事を正しく `skipped` にしつつ、44件が正常に取り込まれることを確認した。
