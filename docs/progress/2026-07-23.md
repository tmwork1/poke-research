## Windows上での `npm run build` 失敗と回避（重要・再発しうる問題）

`npm run build`が、コード変更を一切含まない状態でも次のエラーで失敗することが判明した。

```
[rolldown:vite-resolve] plugin `rolldown:vite-resolve` threw an error
Caused by:
    Tsconfig not found astro/tsconfigs/strict
```

### 原因

Astro 7 / Vite 8 が内部で使うRust製バンドラ Rolldown の、**Windows環境に固有の未修正バグ**（[rolldown/rolldown#8732](https://github.com/rolldown/rolldown/issues/8732)、tsconfig探索時にパス区切り文字（`/`と`\`）が混在し解決に失敗する）。CI（Linux）はこの影響を受けず、実際に直近のCI実行はすべて成功している。Docker Desktop（`node:22-slim`コンテナ）内で同じコードを`npm ci && npm run build`したところ完全に成功することも確認済み。

このバグはビルドパイプライン内の複数箇所（`resolve.tsconfigPaths`によるSSRビルド時の解決、静的ページのプリレンダー時の各ファイルごとのtsconfig自動探索、`builtin:vite-transform`によるTS変換）で個別に発生する。

### 対応（`astro.config.mjs`）

以下の2箇所は`astro.config.mjs`の`vite`設定で回避できた（本プロジェクトは`tsconfig.json`に`paths`エイリアスを一切使っていないため、機能的な影響はない）。

```js
vite: {
  resolve: { tsconfigPaths: false },
  build: { rollupOptions: { tsconfig: './tsconfig.json' } },
},
```

3つ目（`builtin:vite-transform`、Rolldownネイティブプラグイン）は`vite.oxc.tsconfigRaw`での回避を試みたが効果がなく、`@astrojs/cloudflare`アダプター内部の別のビルド呼び出しが`astro.config.mjs`の`vite`設定を継承していないと推測される。この3つ目の発生源・回避策は未特定のまま。

### 今後のローカル検証方針

- `npm test`（ユニットテスト）・`astro dev`（デフさーバー）は上記2つの回避策により、このマシン上でもWindowsネイティブのまま問題なく動作する。
- `npm run build`のローカル最終確認は、Docker Desktop（このセッション中に新規インストール・起動）のLinuxコンテナ経由で行う方針にした。`node_modules`はホストとの共有ではなく名前付きボリューム（`docker run -v <project>:/app -v <volume名>:/app/node_modules`）にマウントすることで、コンテナが入れるLinuxネイティブバイナリ（`@rolldown/binding-linux-x64-gnu`等）でホスト側のWindows用`node_modules`（`@rolldown/binding-win32-x64-msvc`等）を上書きしないようにする（一度誤って上書きし、`npm ci`で再インストールして復旧した）。
- 本番へのマージ判断自体は、これまで通りGitHub ActionsのCI（Linux）を正とする。

## ローカルSupabaseでの実データ動作確認

以前別マシンで使っていた「本番Supabaseとは別の、Docker上のローカルSupabase CLIスタック」という開発環境を、このマシンにも新規構築した（リポジトリ外の`C:\Users\tmtmh\Documents\test\supabase`、`supabase init`→`supabase start`）。全27マイグレーションを適用し、`.env`/.dev.vars`をこのworktreeに配置（本番用の`.env.production`とは別に管理）。

### Windowsネイティブ`astro dev`でも同じRolldownバグを確認

ネイティブ`astro dev`でホームページにアクセスすると、`astro.config.mjs`の回避策を適用済みにもかかわらず500エラーになった。ログを確認すると、astroの標準ロガーが`process is not defined`で例外を握りつぶし本来のエラーが見えない問題（2026-07-05に既知）に加え、実際の原因は**4つ目のRolldownバグ発生箇所**（`Layout.astro`のインラインscriptタグのdev-mode transform、`[TSCONFIG_ERROR] ... src/layouts/Layout.astro?astro&type=script...`）だった。ビルド時と同様、Docker（`node:22-slim`）内で`astro dev`を実行することで回避した。

### Dockerコンテナからホストのローカルsupabaseへ接続する際の追加対応

- `SUPABASE_URL`/`DATABASE_URL`の`127.0.0.1`を`host.docker.internal`に置き換えた別envファイルを、コンテナの`.env`/`.dev.vars`にのみ上書きマウントする方式にした（ホスト側の実ファイルは触らない。`docker run -v <ファイル>:/app/.dev.vars:ro`）。
- `node:22-slim`は`ca-certificates`が入っておらず、GitHub/OpenAI APIへのTLS接続が`unable to get local issuer certificate`で失敗した。`apt-get install ca-certificates`をコンテナ起動時に実行して解決。

### 判明した別問題: ローカルSupabaseの`anon`/`authenticated`ロールにSELECT権限が無い

マイグレーション適用直後、ホームページ（`items`関連のカタログ表示）が`permission denied for table item_tags`（42501）で失敗した。`migrations/027_enable_rls.sql`はRLS POLICY（`TO public USING (true)`）は設定しているが、**テーブルレベルのGRANT（`GRANT SELECT ... TO anon`）を含んでいない**ことが原因と判明した（`\dp`で確認すると、items/item_tags/tags/sourcesいずれも`anon`権限に`r`（SELECT）が無く、`Dxtm`のみ）。Postgresの権限モデルでは、RLSポリシーはロールが元々持つ権限を「絞り込む」だけで、テーブルへの基本権限が無ければRLSポリシーに関わらず拒否される。

2026-07-16の検証では本番・共有ローカル環境の両方でこの権限が機能していたが、それは**マイグレーションファイルに記録されていない、過去に手動で行われたGRANT操作**が既に適用済みだったためと考えられる（全migrationsファイルを`grep`した限り、テーブルへの`GRANT SELECT`文は一切存在しない）。今回のようにマイグレーションだけを適用した完全新規のSupabaseでは再現しない。

ローカルテスト用に`GRANT SELECT ON ALL TABLES IN SCHEMA public TO anon, authenticated;`等を手動で実行し回避した。この問題はGitHubタブ機能とは無関係のため、ユーザー判断で別ブランチ`fix/grant-public-read`（`migrations/028_grant_public_read.sql`）に切り出して対応した。調査の過程で`service_role`にも同種の欠如があることが判明している。詳細は該当ブランチの進捗ログを参照。
