# 進捗メモ 2026-07-07

## 今日の作業

ユーザーから「はてなブックマークからも収集したい」と依頼された。過去（[2026-07-04](2026-07-04.md)・[2026-07-05](2026-07-05.md)）に一度検討し、「はてなブックマークの検索RSSはブックマークされた全ウェブ横断の検索であり非公式寄りの利用になる」という理由で見送り、代わりにBrave Search API経由の個人ブログ収集（`blog.ts`）を採用した経緯があったため、この経緯をユーザーに提示して確認した。ユーザーは「キーワード横断検索（前回見送り案の再採用）」を選び、ToSの懸念は「経緯を承知の上でそれでも進める」と回答した。

### 収集精度の実地検証（実装前の事前調査）

計画確定のため、実装前に `b.hatena.ne.jp/search/text`・`/search/tag` の検索RSSを複数のキーワード（`ポケモン`単独、`ポケモン API`、`ポケモン+API`、`"ポケモン" "API"`、`ポケモン 実装`、`pokeapi`、`ダメージ計算 実装`）で実際にcurlし、生の結果を目視した。

- nonsenseクエリ（`asdkfjhaslkdfjhqwerty12345`）では0件になるため、検索自体は機能している。
- しかし**複数語を組み合わせても実質AND検索にはならない**。「ポケモン API」で検索しても、Anthropic/Claude Fable 5のニュース、AWSブログ、TypeScriptの技術記事など、ポケモンに一切言及していない話題性の高い最近の記事が上位に大量に混入した。クォート（`"ポケモン" "API"`）やプラス結合（`ポケモン+API`）でも同じ結果だった。
- 単独の「ポケモン」検索・タグ検索（`/search/tag`）でも、上位はポケモングッズ・アニメ・カードゲーム相場・育児エピソードなどで占められ、技術記事はほぼ皆無だった。
- 比較的まれな英単語「pokeapi」単独でも、Claude Code README記事・TypeScript Result型記事・Cursor iOSアプリ記事など無関係な最近の技術記事が混入した。
- `&ctgy=it`のようなカテゴリ絞り込みパラメータも効果が無かった（同じ結果セットが返る）。

これは `note.ts` で過去に確認した「母集団の性質上の問題」（[2026-07-05](2026-07-05.md)）と同種だが、対象が全ウェブのためnoteより悪化している。

### 判断: note.tsの前例に倣い実装・運用する

本プロジェクトには、この種の低精度母集団を許容してきた前例がある。note.ts はキーワード検索の生結果が「ほぼ0%が技術記事」と判明した後も、既存のAIレビュー（`article-ai.ts`、主題判定→技術判定の2段階）を安全網として収集を継続する方針を取り、クエリ側の対応は見送っている。この前例に倣い、**収集自体は実装・稼働させ、AIレビューに精度確保を委ね、原始候補数だけ保守的に絞ってOpenAIコストを抑える**方針を採用した。

ユーザーは確認直後に就寝し、「実装→レビュー→最適化→PR→マージまで自律的に」と指示したため、CLAUDE.mdの「mainへのpush・PRマージは実行前にユーザー確認」ルールをこのタスク限定で解除されたものとして最後まで自律実行した。

## 実装内容

`src/lib/importers/blog.ts`（Brave Search経由の個人ブログ収集）をテンプレートとして再利用し、差し替えたのは「候補発見（discovery）」部分のみ。本文抽出・AIレビュー・DB upsert・タグ同期のロジックはそのまま流用した。

- **新規 `src/lib/importers/hatena-feed.ts`**: はてなブックマーク検索RSS（RSS 1.0 RDF形式）の軽量パーサー。依存追加はせず、`<item rdf:about="URL">...</item>` ブロックを正規表現で抽出する純粋関数（`parseHatenaSearchRss`）と数値文字参照デコード（`decodeXmlEntities`）を実装。`cloudflare:workers`非依存のため`tests/hatena-feed.test.ts`で直接ユニットテストできる。
- **新規 `src/lib/importers/hatena.ts`**: `syncHatenaCollection`/`resolveHatenaSyncOptions`。discoveryは`POKEMON_KEYWORDS`（`keywords.ts`、変更なし）の各キーワードで検索RSSを1リクエストずつ取得し、`b.hatena.ne.jp/robots.txt`の`Crawl-delay: 5`に従いリクエスト間に5秒待機する。本文抽出・upsert・タグ同期は`blog.ts`から`extractPage`/`fetchCandidatePage`/`hashBody`/`resolveBlogSource`をexport追加して再利用（`blog.ts`側の変更は`export`キーワード追加とuserAgent引数化のみで、既存の挙動・呼び出しは変更していない）。発見したURLには既存の`isExcludedBlogDomain`（qiita/zenn/note/github等）をそのまま適用し、他インポーターとの重複収集を避ける。新規ソースのため候補数はキーワードあたり既定15件（`MAX_CANDIDATES_PER_KEYWORD`、`HATENA_MAX_CANDIDATES`で上書き可）に絞り、6キーワード×15件=最大90件/日の本文取得・AIレビューを上限とした。`sources.type`は`'blog'`のまま（はてなブログ等の既知プラットフォームはBrave経由発見分と同じsource行を共有）、provenanceのみ`{ source: 'hatena-bookmark-importer', keyword, fetched_at, hatena_bookmarked_at }`で区別する。ジョブ識別（`import_runs.provider`）には自由文字列`'hatena'`を使用（マイグレーション不要）。
- **`src/worker.ts`/`wrangler.jsonc`**: `HATENA_CRON = '30 23 * * *'`（JST 8:30）を追加し、他ジョブと同様のcron分岐・失敗時Webhook通知を実装。
- **新規 `src/pages/api/import/hatena.ts`**: 他インポーターと同型のAPIルート（GET: 既定値確認、POST: `runAndRecord('hatena', 'api', ...)`）。
- **新規 `scripts/collect/collect-hatena.mjs`**: 薄いPOSTラッパー（`npm run collect:hatena`）。
- **`scripts/collect/backfill.mjs`**: `hatena`ターゲットを追加したが、精度検証中のため既定の`BACKFILL_TARGETS`には含めていない（明示指定時のみ対象）。
- **新規 `scripts/eval/eval-collection-hatena.mjs`**: 収集クエリ精度の評価スクリプト（`npm run eval:collection:hatena`）。DB・APIキー不要で、AIレビュー前の生の検索RSS結果を出力する。`eval:all`のデフォルト実行にも組み込んだ（Brave経由のブログ評価と異なりAPIキー課金が無いため）。
- **テスト**: `tests/hatena-feed.test.ts`（新規7ケース）でRSSパース・エンティティデコードを検証。
- **ドキュメント**: `docs/reference/scripts.md`・`README.md`（はてなブックマーク収集ジョブ節を新設）・`docs/development-roadmap.md`（M3拡張リストに追加）・`package.json`（`collect:hatena`/`eval:collection:hatena`エイリアス）を更新。

## 記事カード・検索ページのUI改善（要望5件）

ユーザーから箇条書きで5件のUI改善要望を受け、`feat/item-card-ux-and-source-counts`ブランチで実装した。

1. **タイトルの下線削除**（要望は「アンダースコア」だったが、確認したところ実際は下線＝`text-decoration`の意）: `ItemCard.astro`のタイトルは`<a>`と`<p>`の出し分けだったが、2の対応でタイトル自体を常に`<p>`（非リンク）にしたことで解消した。
2. **タグ以外クリックで元記事へ**: カード全体を「stretched link」パターン化。`.card`に`position:relative`を付け、カード末尾に透明な`<a class="card-link">`（`position:absolute; inset:0; z-index:1`）を追加。タグ・共有ボタン・ブックマークボタンは`z-index:2`で手前に出し、独自のクリックを維持。既読トラッキングも旧`a.item-title`から`.card-link`のクリックへ付け替えた。
3. **タグ: クリックで検索、ホバー/長押しで解説**: タグを`<span role="button">`から実リンク（`href="/tags/{tagName}"`）に変更。解説ポップオーバーは`mouseenter`/`focus`（250ms遅延）と`touchstart`長押し（500ms）でのみ表示するようにし、長押し直後のクリックだけ`preventDefault`して意図しない遷移を防いだ。
4. **ソースボタンに件数表示・多い順ソート**: `migrations/019_add_source_item_counts.sql`でRPC `source_item_counts()`を追加（`link_status != 'broken' AND ai_accepted`の可視条件はtop_tagsやqueryCatalogItemsと揃えた）。`catalog.ts`の`fetchCatalogSources()`をこのRPCと結合し、件数降順でソートして返すよう変更（戻り値型を`Source[]`から`SourceUsage[]`へ）。
5. **圧縮表示ボタンの配置**: `CompactToggle`をeyebrow行から、検索結果グリッド直前の上側`Pagination`と同じ行（`.top-controls`、`position:relative`＋`.compact-toggle-slot`を`position:absolute; right:0`で右端固定）に移動した。
