# 2026-07-16

## `/items` の範囲外page指定でPGRST103エラー（修正済み・PR #78）

Discordの運用アラートで `/items` の500エラー（`PGRST103`: `offset 20 requested, but there are only 16 rows`）を検知。`page`クエリパラメータに上限チェックが無く、記事削除等で総件数が減った後に古い`page`（クローラのキャッシュ等）でアクセスされると、`.range()`のoffsetが総件数を超えPostgRESTが例外を返していた。

`src/lib/catalog.ts`の`queryCatalogItems`で`PGRST103`を検知した場合、件数のみ取得するクエリ（head:true）にフォールバックし、空リスト＋正しい総件数を返すよう修正（`fetchCatalogItemsPage`を経由する`/items` `/tags/[name]` `/papers` `/`すべてに適用）。

## Supabase RLS未設定（重大セキュリティ指摘）への対応

Supabaseから「`rls_disabled_in_public`: publicスキーマの全テーブルでRLSが無効」という重大アラートを受け調査。全マイグレーション（001〜026）にRLS/POLICYの設定が一切なく、DB書き込みが実質すべて匿名キー（`SUPABASE_PUBLISHABLE_KEY`）1本で行われ、管理者API（Basic認証）・cronジョブの認可がアプリ層のみでDBには伝播していない状態だった。特に`bookmarks`（他人の読み書き）・`users`（他人のメール閲覧）が実害の大きい経路。

### 対応方針

1. `src/lib/supabase.ts`に`getSupabaseAdminClient()`（`SUPABASE_SECRET_KEY`、RLSを常にバイパスするservice_role）を新設し、管理者API・cronジョブ・インポーター等の書き込み経路（`db.ts`/`audit.ts`/`import-runs.ts`/`importers/common.ts`/`importers/link-check.ts`/`tag-explain.ts`）を全て切り替え。
2. ブックマーク機能（`bookmarks.ts`・`catalog.ts`のブックマーク関連4関数）は、既存の`createUserSupabaseClient`（Cookieベースのユーザーセッション）を使うようシグネチャを変更し、`auth.uid()`ベースのRLSが正しく効くようにした。呼び出し元8箇所（`index.astro`/`items/index.astro`/`papers/index.astro`/`tags/[name].astro`/`mypage.astro`/`api/bookmarks/index.ts`/`api/bookmarks/[itemId].ts`）を更新。
3. `migrations/027_enable_rls.sql`で全publicテーブルにRLSを有効化。items/sources/tags/item_tags/annotationsは全行SELECTのみ公開（書き込みはadminのみ）、bookmarks/usersは`auth.uid()`で本人行に限定、audit_logs/import_runs/feed_subscriptions/ai_prompt_hashesはポリシー無し（service_roleのみ到達）。`bookmark_counts`・`sync_items_bookmarks_count`はRLSをまたいで動作する必要があるため`SECURITY DEFINER`を付与。

### 検証

- CI（`.github/workflows/ci.yml`）の`migrations`ジョブは素のpostgres:16に対して素朴なDDL検証のみ行っており、`auth.uid()`関数や`anon`/`authenticated`/`service_role`ロールが存在しないため、このままでは027が失敗する。stub手順に role/`auth.uid()`の作成を追加した。Dockerで同条件のコンテナを立てて実際に全マイグレーションが通ることを確認済み（RLS有効化・ポリシー・SECURITY DEFINERの付与も`pg_class`/`pg_policy`/`pg_proc`で確認）。
- ローカル共有Supabase（project-id "supabase"）にも027を適用し、PostgREST経由で匿名キーの公開SELECT成功・書き込み拒否（42501）・bookmarks/users/audit_logsの非公開確認・service_roleキーでのバイパス確認まで実施。
- `astro dev`でアプリを実際に起動し、一覧（`/` `/items` `/papers` `/tags/[name]`）・マイページ・ブックマーク追加/一覧/削除・管理者API（sources作成/削除、audit_logs記録確認）・タグ解説APIを一通り動作確認。
- ローカル`.dev.vars`に`SUPABASE_URL`/`SUPABASE_PUBLISHABLE_KEY`はあったが`SUPABASE_SECRET_KEY`が無く、`astro dev`（Cloudflare Workersのローカル実行環境）は`.env`のprocess.envフォールバックを拾わないため、追加が必要だった（`scripts/db/*.mjs`のような素のNodeスクリプトとは異なる挙動）。
- `npm test`（129件）・`npm run build`とも成功。

### 教訓

- Supabaseの匿名キー（`SUPABASE_PUBLISHABLE_KEY`）は「サーバー内にしか無いから安全」ではなく、RLSが無い限りDBへの直接到達を防ぐ層が存在しない。管理者APIのBasic認証やcronの実行制御はアプリ層の認可であり、DBそのものは無防備なままだった。
- CIの`migrations`ジョブは素のpostgres:16を使っているため、Supabase固有のロール（anon/authenticated/service_role）や関数（`auth.uid()`）に依存するDDLは、CI側にも同等のstubを用意しないと検証できない（[[cloudflare-cron-trigger-limit]]同様、CI/本番の環境差に起因する落とし穴）。
